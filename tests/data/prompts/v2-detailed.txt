# 核心规则

**段格式（重要）**
- 每个叙事段只能是以下两种类型之一，严格禁止混合：
  - 旁白段：纯叙述描写，不含任何角色对话。一段只说一件事，15-40 字。
  - 对话段：纯角色对话。格式固定为 `角色名: 对话内容`（英文半角冒号，冒号后空一格，对话不加引号）。
- 角色的动作、表情、语气描写单独成旁白段，不得塞进对话段。
- 每段不超过 50 字，超过则拆为多段。段之间空行分隔。

**编号**
- 叙事段前加数字编号，从 1 开始。所有 narrative 块（含不同分支）共享同一序列。
- 不同分支块的编号必须连续——后出现的分支接续前一分支的编号，禁止重复使用已出现的编号。
- 选项行用字母编号（A / B / C / D），不占数字序列。

**options（重要）**
- 第一行必须是 `choice: 变量名`（如示例 `choice: approach`）。此行不展示给玩家，仅用于程序识别。
- state 条件中通过此变量名 + 数字引用玩家的选择（A=1, B=2, C=3, D=4），如 `if approach == 1`。
- 禁止使用 `choice == "A"` 或 `选择1 == 1` 等占位写法——变量名必须与 `choice:` 声明完全一致。

**段数与 bridge**
- 本轮生成 60-100 个叙事段。bridge 放在总段数的约 40% 处——总段数 × 0.4，向下取整。
  例如：总 80 段 → bridge 在第 32 段后（32 = 80 × 0.4）。总 100 段 → bridge 在第 40 段后。
- bridge 是交互区块与尾部叙事的分界线：bridge 之前放 options、state、checkpoint，bridge 之后只能有 narrative 区块。
- bridge 之前的 options 和 state 必须使用 `:main` 分支（如 `--- options:main ---`、`--- state:main ---`）。
  命名分支（如 `:take_lead`）只能出现在 bridge 之后的 narrative 区块中。
- `bridge` 是保留字，禁止用作分支名（如 `--- narrative:bridge ---` 是错误的）。

**state 语法**
- 无条件变更：`@var 变量名 操作符 值`
- 条件变更：`if 条件 -> @var 变量名 操作符 值`
- number 类型：`+N`（增加）/ `-N`（减少）/ `=N`（设为）
- string 类型：`=值`（设为）
- list 类型：`+元素`（追加）/ `-元素`（移除）
- 条件可引用 choice 变量名（`if approach == 1`）或状态变量（`if 信任度 >= 30`）。用 `and`/`or` 组合。

**checkpoint（重要）**
- `node {node_id}` 或 `end`，必须附 `summary:`（1-2 句的中文摘要）。
- 分支（如有）：`if 条件 -> route {target_node_id}`。放在 bridge 之前。
- node_id 和 route 目标必须严格复制大纲中列出的节点 ID，不得修改、拼接后缀或自创。
  例如大纲中有 `ch2_confrontation`，则必须写 `node ch2_confrontation`，禁止写 `ch2_confrontation_resolved` 或 `ch2_final`。

**禁止清单**
- bridge 之后出现 options、state、checkpoint 等任何非 narrative 区块
- bridge 之前的 options/state 使用非 `:main` 的分支名
- narrative 正文中 `---` 单独成行（会被误解析为区块分隔符）
- 用 `bridge` 或其他保留字作为 narrative 分支名
- 一段内同时包含旁白和对话（混排）
- 对话段使用引号包裹对话内容
- 对话段用代词代替角色名（如 `你: ...`、`我: ...`。必须使用声明的角色姓名，如 `林焰: ...`）
- 对话段中附加动作描写（如 `他笑了笑: ...`，应拆为旁白 `他笑了笑。` + 对话 `角色名: ...`）
- JSON、XML 替代区块标记
- 对玩家说话（"你选择了……"、"请选择……" 等打破第四面墙的表述）
- 引用不存在的变量名或大纲节点 ID

# 质量要求

每段只做一件事——描写一个画面或表达一句对白。对话与旁白交替出现，避免连续 3 段以上纯描写。选项后果在叙事中铺垫。bridge 之后制造悬念，让玩家期待下一轮的展开。
