## 摘要

交互式文字游戏的传统实现依赖预编写的内容分支，难以提供真正开放式的玩家体验。大语言模型（LLM）具备实时生成叙事内容的能力，但其推理延迟与交互实时性之间存在结构性矛盾。本文介绍 Storyloom——一个 AI 驱动的交互式文字小说游戏引擎，它通过三项核心设计的协同——**桥接机制**（以叙事缓冲掩盖 LLM 推理延迟）、**流式解析**（以逐行处理替代全量等待）、**时序模型**（以首字可展示时间替代总生成时间作为核心指标）——在工程层面解决了这一矛盾。本文系统阐述了项目的 9 项设计理念及其理论依据，记录了从实验验证到工程落地的完整实践过程，分析了项目与《程序设计》《数据结构》课程知识点的深层联系，并总结了人机协作开发中的方法论经验。项目全程无工业参考，所有关键设计均独立构思并通过实验验证。当前 Phase 1 已完整实现：包含共创流程、叙事循环、存档系统、结局检测等完整引擎功能，以及基于 FastAPI + SSE 的 Web 界面和跨平台打包发布。

## 1. 背景

### 1.1 项目起源

本项目并非任何原有课程作业的产物，而是个人一直以来的想法——**AI 实时生成式游戏**。

当前 AI 在内容生成领域的主流应用模式是"一次性生成"：用户提交需求 → AI 完整产出全部内容 → 用户在生成结果上进行有限的、传统的交互。这种模式将 LLM 定位为"内容工厂"——AI 的价值集中在"生成"环节，而"交互"环节与生成过程完全脱节。无论是 AI 写作助手、AI 代码补全还是对话机器人，绝大多数应用都遵循这一范式。

本项目的核心洞察在于：**"实时生成"可以运用于更广泛的、需要频繁用户交互的场景**。在游戏叙事中，如果 AI 能够在玩家每一次选择后即时生成接下来的故事内容，那么游戏体验将从根本上被重塑——不再是"从预生成的分支中挑选"，而是"与 AI 共同编织独一无二的叙事"。这一理念将 LLM 从幕后的"内容生产者"提升为前台"实时的、参与式的叙事协作者"。

### 1.2 演进目标

项目的长期愿景规划为三个阶段（<!-- UPDATE: Phase 状态随开发进展更新 -->）：

- **Phase 1（✅ 已完成）**：AI 驱动的文字小说冒险游戏。LLM 作为"说书人"，程序负责流程管理、状态校验与上下文管控。完整的引擎核心 + Web 界面 + 打包发布。
- **Phase 2（规划中）**：AI 驱动的视觉小说游戏。在文字叙事基础上引入静态背景、角色立绘、场景切换等视觉元素，提升沉浸感。
- **Phase 3（远期愿景）**：3D AI 实时生成游戏。AI 实时生成 3D 场景、角色动画与交互内容，实现真正意义上的"无限世界"。

### 1.3 项目特殊性

本项目最显著的特征是：**在设计理念和工程方案上，没有现成的参考模板**。

从提示词约束体系、LLM 返回的 XML 格式规范，到桥接机制、时序模型、流式解析——每一处核心设计都是从头独立构思、反复实验验证、最终落地实现的。所有关键决策均基于第一性原理推演和实验数据，而非对既有方案的模仿或改良。

这一方面带来了极大的设计自由，另一方面也意味着每一个决策都需要承担"无人验证过"的风险。这份报告中详述的每一个设计理念，背后都经历了反复的推敲、辩论和实验。

## 2. 游戏流程演示

> 在深入设计理念之前，本节概述 Storyloom 的完整用户体验流程，为后续技术讨论建立直观的认知基础。

Storyloom 的完整用户体验分为五个阶段。

### 2.1 主菜单

用户启动程序后进入 Web 主菜单界面，提供以下功能入口：

- **语言设置**：支持简体中文（zh-CN）、繁体中文（zh-TW）、英文（en），运行时即时切换。
- **API 配置**：设置 LLM API 密钥、Base URL 和模型名称。配置持久化至本地 `config.json`。
- **开始新游戏**：进入共创流程。
- **继续游戏**：一键恢复最近游玩存档。
- **载入存档**：浏览全部游戏及存档点，选择恢复。
- **制作人员**：项目信息展示。

### 2.2 共创阶段（Co-Creation）

这是一个完全自由的聊天平台。用户输入原始故事想法（一句话到一段话均可），AI 接管对话，围绕世界观、主角设定、故事基调、核心冲突、故事篇幅等维度展开多轮问答。用户随时可以决定"开始生成"——不限制问答轮次，完全由用户把控讨论深度。

生成阶段由一次 LLM 调用完成三项输出：
1. **故事设定（story_config）**：题材、篇幅档位、故事标签、主角信息、基调、冲突、角色关系等。
2. **游戏变量（variables）**：最多 3 个自定义状态变量（≤2 个数值型 + ≤1 个标签型），用于跟踪关键叙事状态。
3. **故事大纲（outline）**：有向图结构，定义章节节点（5-30 个，视篇幅档位）、每个节点的叙事目标、节点间的条件路由。

### 2.3 叙事阶段（核心玩法）

共创完成后进入叙事循环。每轮流程如下：

1. AI **实时流式生成**带选项的故事文本（XML 结构化输出），内容逐字出现在屏幕上。
2. 文本到达 `<bridge/>` 标记后，**程序在后台立即发起下一轮 API 请求**，前台继续展示桥接后的叙事内容。
3. 当前轮文本展示完毕，玩家面对 2-4 个选项（部分选项可能因状态条件不满足而灰显）。
4. 玩家做出选择 → 程序验证并更新游戏状态 → 展示选择后果叙事 → 下一轮自动衔接，**无感知延迟**。

### 2.4 结局与冒险日志

当故事大纲的最后一个节点（路由为空的结局节点）被触发时，游戏进入结局。此时：

- 绿色结局选项替代常规选项，玩家点击后进入冒险日志页面。
- 冒险日志由一次独立的 LLM 调用生成，以 Markdown 格式回顾整个故事的关键节点和玩家选择，约 500-1000 字，支持下载导出。

### 2.5 存档系统

- **自动存档**：每次在 checkpoint 节点推进时自动触发，追加式保存，不覆盖历史存档。
- **读档管理**：按游戏分组浏览，每局游戏下展示全部存档点，支持加载和删除。
- **继续游戏**：O(1) 恢复最近游玩位置。

## 3. 重要设计理念

> 本章是报告的核心。上述游戏流程的每一个环节——从 LLM 按照精确格式生成叙事文本，到玩家感知不到轮次边界的无缝体验，再到程序在 AI 创造力与系统确定性之间的权衡——背后都对应着一组精心设计的技术理念。这些理念并非 AI 构想的具体实现方式，而是在反复思考、实验和辩论后确立的**设计原则和理论框架**。下文按逻辑依赖关系（而非重要程度）排序：提示词设计是基础，桥接机制与流式解析是体验核心，时序模型是理论支撑，其余理念构成完整的工程保障体系。

### 3.1 提示词设计规范

#### 3.1.1 核心地位

提示词（Prompt）是项目最核心的基础。LLM 的所有输出都依赖提示词的引导和约束——提示词质量直接决定了故事内容的品质、输出格式的合规率，进而影响整个游戏体验的稳定性。如果提示词不够精准，下游的解析器、状态校验、上下文管理都会因为输入质量不稳定而频繁出错。

#### 3.1.2 迭代历程

提示词设计经历了三个主要阶段：

| 阶段 | 格式方案 | 问题 |
|------|---------|------|
| 初期探索 | 固定约束 Markdown 格式 | LLM 频繁偏离格式，无法可靠解析 |
| 中期改进 | `--- block ---` 文本块分隔 + 叙事段编号 | 正确率仅 20-74%，bridge 位置不稳定 |
| 最终方案 | XML 结构化输出 + `NNN|` 逐行编号 | 正确率 100%（Frame-v1 3/3） |

关键转折点是发现 **XML 是 LLM 预训练数据中的"母语"**——LLM 对 XML 标签的嵌套结构和属性语法有极强的模式识别能力，远优于自定义文本格式。这一发现来自反复实验中的观察，而非预先假设。

#### 3.1.3 提示词 v4 模板与七项约束有效性原则

提示词模板 v4 经历了 6 轮迭代、30+ 次测试，最终总结出 **七项约束有效性原则**。这些原则是项目最重要的方法论输出之一：

1. **反例约束（Counter-Example Constraints）**：不仅告诉 LLM 该做什么，还展示典型错误。例如在格式示例中故意展示编号跳号、bridge 位置不当等反例，并标注为"错误"。

2. **正反双重覆盖（Positive-Negative Dual Coverage）**：每项关键约束同时出现在正面规则和反面禁止中。仅靠"应该做 X"可能被忽略，同时加上"禁止做非 X"形成双重保险。

3. **注意标签（Attention Labels）**：对高频错误规则附加 `(important)` 标记。这不是修辞——实验表明明确的注意提示能显著降低特定类型错误的出现率。

4. **示例-规则屏障（Example-Rule Barrier）**：在格式示例结束后加入显式提醒（"以上为示例，以下为规则"），防止 LLM 将示例中的编号延续到正式输出中。这是针对 LLM"惯性延续"问题的专门对策。

5. **具体优于抽象（Concrete over Abstract）**：使用具体的数字和案例传达约束，而非使用比例或模糊描述。例如直接给定行数范围 150-300 而不是"适当长度"。

6. **显式禁止优于隐式模式（Explicit Prohibition over Implicit Pattern）**：独立的 `Forbidden` 小节集中列出所有禁止事项，不与规则部分混排。实验表明集中式的禁止列表被 LLM 遵守的概率高于分散在各处的"不要"。

7. **关键点不省 Token（Don't Skimp on Critical Points）**：对频繁出错的规则投入更多 token 展开说明。在 token 预算有限的前提下，策略性倾斜而非均匀分配。

#### 3.1.4 实验验证

提示词 v4 模板的实验数据（6 轮迭代，30+ 次测试）：

| 指标 | 优化前（v3） | 优化后（v4） | 提升 |
|------|------------|------------|------|
| 格式正确率 | 33% | 83% | 2.5× |
| 平均 TTFT | 38s | 11s | 3.5× |
| XML 格式正确率（Frame-v1） | — | 100%（3/3） | — |

**跨题材验证**：使用 4 种不同题材（奇幻、科幻、武侠、都市）各进行 3 轮测试，发现并修复了两个跨题材共性问题——bridge 位置偏差和选项置于 bridge 之后的倾向。

**Segment 长度-TTFT 实验**：为确定最优输出长度和 bridge 位置，设计了 4 个段数梯度，每种 3 次运行，测量 TTFT、实际段数和格式正确率（8 项 XML 检查）：

| 档位 | 段数范围 | 平均 TTFT | 平均段数 | Bridge 位置 | 正确率 |
|------|---------|-----------|---------|------------|--------|
| T1（对照） | 60-120 | ~48s | 84 | ~40% | 2/3 |
| T2 | 120-200 | ~52s | 156 | ~55% | **3/3** |
| T3 | 180-280 | ~56s | 218 | ~62% | 2/3 |
| T4 | 240-360 | ~58s | 285 | ~70% | 1/3 |

关键发现：
- **段数增加 3×（T1→T4），TTFT 仅增加 ~20%**——远非线性关系，证明**提示词大小（输入 token 数）是 TTFT 的主要影响因素，而非输出长度**。
- T4 的正确率明显下降——LLM 在超长输出时难以维持格式一致性。
- **最优范围：T2（120-200 段）**——正确率最高（3/3），TTFT 可控（~52s），缓冲文本充足。最优 token 预算：**12,288 tokens**。

实验结束后，配置参数在代码和规范文档中持续迭代：Phase 2 在 T2 段数下进一步调整 bridge 位置比例，最终收敛于 `BRIDGE_POSITION_RATIO = 0.75`、`MIN_TAIL_LINES = 25`，并通过后续的 gameplay 测试验证了参数的实际效果。

#### 3.1.5 对话式消息数组架构

提示词架构最大的结构性变革是从"每轮发送完整 System Prompt"迁移到**对话式消息数组（Conversational Message Array）**：

- **Round 1 永久锚定**：首轮用户消息包含全部格式规范 + 故事上下文 + 完整 XML 示例。此消息**永不压缩、永不删除**。
- **Round N 轻量上下文**：后续轮次仅发送大纲进度、当前状态、桥接文本和可选的错误反馈，不重复格式规范。
- **滑动窗口 + 压缩**：保留最近 3 轮完整对话，更早轮次压缩为 checkpoint 摘要对。

这一架构消除了每轮 ~3,000 token 的格式重复开销。

#### 3.1.6 提示词体系全景

Storyloom 在四个阶段与 LLM 交互，每个阶段使用不同结构的提示词。理解这些提示词的构成，是理解整个系统如何"驾驭"LLM 的关键。

**阶段一：共创对话**

这是整个游戏流程的起点。系统向 LLM 发送一条 system 消息，定义其角色为"故事共创伙伴"——任务是通过对话收集信息，而非直接生成故事。消息包含：角色声明、语言指令（从 `lang_meta/{lang}.json` 注入）、五项探索维度（故事篇幅、世界观、主角、基调、冲突），以及三条硬性禁止（不生成故事内容、不固定问题数量、不单方面总结）。玩家与 LLM 在这个框架内进行多轮自由对话，直到玩家决定推进到下一步。

**阶段二：共创生成**

玩家确认讨论充分后，系统在同一对话中追加一条 user 消息，要求 LLM 一次性产出三项结构化内容。消息结构如下：

| 消息段落 | 内容 |
|---------|------|
| 上下文提示 | "基于以上对话，生成完整的故事设定"——将生成锚定在前面的共创讨论上 |
| 变量规则 | 硬约束：变量总数 ≤3（数值 ≤2、标签 ≤1）。附题材-变量种子参考表，引导 LLM 选择有叙事意义的变量 |
| 大纲规则 | 节点数量按篇幅档位（短篇 5-10 / 中篇 10-20 / 长篇 20-30），节点目标为章节弧光而非单一场景，分支条件只能引用已声明变量，最终节点路由必须为空（结局检测依据） |
| 输出格式 | 三个 `=== 标记 ===` 分隔的区块：`story_config`（INI 风格键值，含题材、标签、主角、基调、冲突等 13 个字段）、`variables`（每行 `名称: 类型, 初始值`）、`outline`（`[node]` 块，含 id/title/goal/routes） |

LLM 在一次调用中同时产出故事设定、游戏变量和大纲有向图，程序随后对每个区块进行解析和校验。

**阶段三：叙事循环（每轮）**

这是提示词设计最复杂的部分。叙事阶段的提示词采用**对话式消息数组**架构——不是每轮重复发送完整指令，而是通过消息的增删和压缩管理上下文：

*首轮消息结构：*

系统发送一条由两部分拼接而成的 user 消息：

| 部分 | 内容 | 永久性 |
|------|------|--------|
| **前缀（Prefix）** | 角色声明 → 输出格式规范 → 结构大纲（40 行 XML 骨架示意）→ 七个 XML 元素定义 → **完整格式示例**（45 行奇幻故事，展示全部元素的实际使用）→ 核心规则（段格式、行数/bridge 位置、选项语义、state 变更、checkpoint 规则、XML 规则、9 条禁止项）→ 品质要求 → 故事上下文（世界观、主角、基调、冲突、角色列表） | **永久锚定，永不删除** |
| **轮次模板（Turn）** | 大纲进度树（含完成/活跃/待定标记）→ 当前节点与目标 → 当前状态变量快照 → 错误反馈（如有）→ 输出行数约束 → 桥接文本 | 每轮更新 |

*后续轮次：*

仅发送**轮次模板**（不含前缀）。LLM 通过已锚定的首轮消息"记住"格式规范，无需每轮重复 ~3,000 token 的格式说明。

*上下文压缩：*

当对话轮次超过 `WINDOW_SIZE = 3`（且总轮次 ≥5）时，最早的完整轮次被压缩：提取其中的 checkpoint 摘要信息，合成为一对简短的 user/assistant 消息（"关键事件回顾" / "先前事件的总结"），释放上下文空间。

*叙事阶段的 LLM 返回格式：*

每行以 `NNN|`（3 位补零递增序号）开头，程序解析前自动剥除。根元素为 `<story>`，内含以下元素：

```
001| <story>
002| <seg>叙事段：独白或对话，每段一个信息点</seg>
003| <seg>Kael: 对话以"角色名: 内容"格式</seg>
...
| <branch name="attack">         ← 段内叙事分支
|   <seg>分支变体 A 的叙事</seg>
| </branch>
...
| <choice id="fight_or_talk">    ← 玩家选项（至多一处）
|   <opt key="1" branch="attack">发起攻击</opt>
|   <opt key="2" branch="negotiate" if="魅力 > 50">尝试谈判</opt>
| </choice>
| <set var="勇气" op="+" val="10" if="choice=attack"/>  ← 状态变更建议
| <checkpoint node="ch2_confront" summary="...">   ← 大纲路由
|   <route if="勇气 > 50" target="ch3_heroic"/>
|   <route target="ch3_cautious"/>
| </checkpoint>
...
| <bridge/>                       ← 桥接标记（必须恰好一次）
...
| <branch name="attack">         ← 桥接后纯叙事（无交互元素）
|   <seg>攻击的后续叙事...</seg>
| </branch>
| </story>
```

完整格式示例（45 行带编号的奇幻故事）见 `docs/spec/prompt-design.md` §4。该示例是 LLM 学习输出格式的核心参照——LLM 模仿示例的准确度远高于遵循文字规则。

**阶段四：冒险日志**

结局触发后，系统发起一次独立的 LLM 调用（不纳入叙事对话的消息数组）。提示词结构为：

| 段落 | 内容 |
|------|------|
| 角色声明 | "你是冒险日志作者，为已完成的文字冒险游戏撰写玩家回顾" |
| 故事背景 | 题材、设定、主角、基调、冲突、角色列表 |
| 故事大纲 | 完整大纲树（含各节点的完成标记和摘要），附注说明以摘要（而非大纲目标）为回顾依据 |
| 输出章节 | `## Adventure Recap`（按章节回顾）+ `## Ending`（温暖、满足的结局叙述）+ `## Final State`（各变量的终值和一句话反思） |
| 格式要求 | Markdown 纯文本（无 XML、无分隔符）、第二人称直接面向玩家、500-1000 词 |

四个阶段的提示词各有侧重但共享同一设计哲学：**精确的约束描述 + 完整的格式示例 + 清晰的禁止边界**。完整模板和规范见 `docs/spec/prompt-design.md` 及 `src/storyloom/core/prompt_builder.py`。

### 3.2 桥接机制

> 提示词规范解决了"LLM 输出什么"的问题。桥接机制解决的是"LLM 输出的时候，玩家在做什么"的问题——这是实时生成体验面临的结构性挑战。

#### 3.2.1 问题的本质

AI 实时生成面临一个结构性矛盾：**LLM 推理必然需要时间**。从提交 prompt 到收到首个有意义的内容 token（Time to First Token, TTFT），通常需要 10-30 秒。这是实时生成无法与传统预生成内容+即时交互匹敌的根本原因——玩家不能接受每轮选择后等待半分钟才能看到结果。

一种常见的规避思路是"全预生成"——把 AI 的生成时间集中到开局前一次性承担。但这牺牲了"实时生成"的核心价值：故事无法真正跟随玩家的选择即时演化。

#### 3.2.2 解决方案

桥接机制的核心思路是：**不让玩家等待，而是让等待发生在玩家正在阅读的时候**。

具体机制如下：

1. 每轮 LLM 输出被 `<bridge/>` 元素分为两个区域：
   - **Bridge 之前（交互区）**：包含叙事推进、选项（`<choice>`）、状态变更（`<set>`）、checkpoint 路由。
   - **Bridge 之后（叙事缓冲区）**：纯叙事内容，不包含任何交互元素。受 `current_branch` 控制，仅展示玩家所选分支对应的后续叙事。

2. 流式解析器在读到 `<bridge/>` 的瞬间，**后台 daemon 线程立即发起下一轮 API 请求**，同时前台继续展示 bridge 之后的叙事内容。

3. 当玩家阅读完当前轮的全部内容、做出选择后，下一轮的 API 响应已经（或接近）完成——TTFT 被桥接后的叙事展示时间完全覆盖。

#### 3.2.3 桥接机制的核心价值

桥接机制并非简单的"提前发请求"。它重新定义了叙事轮次的边界：

- **没有桥接**：每轮 = 等待（TTFT）+ 阅读 + 选择 + 等待（TTFT）+ … → 体验被频繁打断。
- **有桥接**：每轮 = 阅读（含 bridge 后内容）+ 选择 + 无缝下一轮 → **与非实时生成游戏相同的体验**。

**Bridge 位置的优化逻辑**是桥接机制与流式解析、TTFT 发现三者协同的典型体现。最初 bridge 设定在输出 40% 处——此时 post-bridge 缓冲文本占 60%，留出大量"安全垫"。但 TTFT 实验揭示了一个关键事实：**提示词大小主导 TTFT，而非输出长度**。这意味着可以大幅增加输出总量而 TTFT 增幅很小，从而为交互区争取更多空间。

配合流式解析——首行在 TTFT（10-30s）即可展示，而非等到全部生成——交互区内容可以立即流式呈现给玩家。这促使 bridge 位置从 40% 前推至 **75%**：交互区从 40% 扩展到 75%，为叙事推进、选项、状态变更和 checkpoint 路由提供了更充裕的空间；post-bridge 缓冲缩减至 25%，配合优化后的 TTFT（~11s），阅读时间（150 行时约 9s、300 行时约 19s）足以覆盖下一轮的等待。`MIN_TAIL_LINES = 25` 确保桥接后每个分支有足够叙事量。

最终效果：**流式解析让玩家尽快看到内容，桥接机制让玩家永远不会等待内容**。两者各自独立设计，但只有同时存在才能实现完整的无缝体验。

### 3.3 时序模型

> 桥接机制解释了"在哪里"掩盖延迟，流式解析解释了"怎么"降低等待。时序模型回答的是一个更根本的问题：**在分布式异步系统中，如何定义和度量"用户体验时间"**。

#### 3.3.1 模型定义

桥接机制和流式解析的背后，是一个更基础的**三层时序模型**。这是在工程实践中总结出的基本原理，为项目全部时序相关设计提供了理论依据：

```
LLM 生成流（T_gen） ≥ 程序解析流（T_parse） ≥ UI 展示流（T_display）
```

三个流各自独立运行，分属不同的系统层：

| 流 | 负责层 | 速度特征 | 可控性 |
|------|---------|---------|--------|
| LLM 生成流 | API 服务端 | token 逐块产出。TTFT 10-30s（首 token），总生成时间 25-50s | 不可控——受模型推理速度和网络延迟制约 |
| 程序解析流 | 引擎（StreamingXmlParser） | 逐行流式解析，与生成流几乎同步（生成即解析，微秒级/行） | 完全可控——纯本地计算 |
| UI 展示流 | UI 层（CLI / Web） | 消费引擎产出的事件，按自身节奏展示（自动模式固定间隔 / 手动模式等待确认） | 可控——设计目标是接近自然阅读节奏 |

**流间同步规则**：

1. 解析流不能快于生成流——否则会因等待 token 而阻塞。通过 `queue.Queue` 天然同步：daemon 线程推入 token，主线程拉取，Queue 为空时自动阻塞。
2. 展示流不能快于解析流——否则会出现"空白段"。通过 UI 队列缓冲实现：引擎产出事件推入队列，展示端按节奏消费。
3. 解析流 ≤ 展示流是**正确性约束**（解析慢了用户会等），生成流 ≥ 解析流是**物理约束**（不可改变）。

#### 3.3.2 Bridge Pre-Fetch 的时序约束

桥接机制的可行性取决于一个时序不等式：

```
TTFT（下一轮） < T_bridge_text（当前轮 bridge 后文本的展示时长）
```

其中 `T_bridge_text = post_bridge_lines × t_per_line × RATE`（RATE 为阅读速度比例，通常 50%）。当这个不等式成立时，下一轮的首 token 在 bridge 文本播完前到达，玩家感知不到等待。

关键推论：
- **如果使用全量解析**：首行可展示时间 = TTFT + 完整生成时间（35-80s），远超 bridge_text 展示时长（9-19s）→ bridge pre-fetch 无效，玩家必然空白等待。
- **如果使用流式解析**：首行可展示时间 = TTFT（10-30s），bridge_text 展示时长（9-19s）可覆盖 TTFT 下界 → 多数情况下无缝衔接。
- **如果 TTFT 被 Prompt 优化缩短**（38s→11s）：不等式更稳定成立 → bridge 位置可前推至 75%，释放更多交互空间。

这就是时序模型指导设计决策的完整逻辑链：**流式解析是 bridge pre-fetch 生效的前提，Prompt 优化是 bridge 位置优化的前提，三者共同构成无缝体验的工程基础**。

#### 3.3.3 核心指标的重定义

传统思维关注"LLM 生成全部内容需要多长时间"（总生成时间，25-50s）。时序模型指出：**正确的核心指标是"首个可展示内容何时出现"（TTFT，10-30s）**。

这一重定义是整个流式解析设计决策的理论基础：
- 如果关注总生成时间 → 必须等全部生成完毕才能开始展示 → 玩家每轮等待 25-50s → 不可接受。
- 如果关注 TTFT → 第一行生成即可开始展示 → 玩家等待降至 10-30s → 配合桥接机制消除感知延迟。

#### 3.3.4 推广价值

三层时序模型不仅适用于本项目。它可以推广到更广泛的 AI 实时生成场景——**任何需要将 LLM 输出实时呈现给用户的应用，都应按照流式解析而非全量解析来设计**。判断标准不是"生成速度"，而是"首字可展示时间"。bridge pre-fetch 的时序约束不等式同样通用：**只要"下一轮等待时间 < 当前轮内容的用户消费时间"，就可以隐藏延迟**。这一原则对 AI 对话产品、AI 写作助手、AI 代码补全等场景具有普遍适用性。

### 3.4 流式解析

> 时序模型确立了 TTFT 为关键指标的理论前提。流式解析是这一理论前提在工程层面的直接推论——也是项目中经历最激烈设计辩论的决策。

#### 3.4.1 设计决策

基于时序模型，项目确定了**流式逐行解析**的架构方案：

- **流式方案**：LineBuffer 接收 SSE token 片段，累积到完整行后立即送入 StreamingXmlParser → 解析为事件 → 推送至 UI 展示。首行可展示时间 ≈ TTFT。
- **全量方案（被否决）**：等待 LLM 完整输出 → 一次性解析全部 XML → 批量展示。首行可展示时间 = 总生成时间（25-50s）。

#### 3.4.2 与 AI 的辩论——人类判断力的关键体现

流式解析的设计决策经历过一次重要的反复，这是项目中人类判断力发挥关键作用的典型案例：

1. **初期设计**：基于时序模型，确定使用 StreamingXmlParser 进行流式逐行解析。
2. **07-10**：AI 辅助开发过程中，AI 建议"流式解析复杂度高、全量解析更简单可靠"，并在一次代码重构中删除了 StreamingXmlParser，替换为全量解析方案。
3. **07-11**：我（项目负责人）审查后发现这一变更是错误的——全量解析忽略了时序模型的核心洞察（关键指标是 TTFT 而非吞吐量），会造成每轮 25-50 秒的额外等待。我坚持恢复了流式解析方案，并从 git 历史中恢复了 StreamingXmlParser 代码（同时修复了 5 个 bug）。
4. **最终验证**：流式解析被证明是完全正确的设计——配合桥接机制，共同实现了无缝叙事体验。

这次辩论的核心分歧在于：**"简单"（全量解析的工程简单性）vs "正确"（流式解析的用户体验正确性）**。LLM 倾向于选择前者——因为它在代码层面评估方案，而非从用户体验第一性原理出发。而人类（项目负责人）能够跳出代码视角，站在玩家体验的角度判断孰优孰劣。

#### 3.4.3 LineBuffer + StreamingXmlParser 架构

```
SSE token stream
    │
    ▼
LineBuffer.feed(text)
    │ 累积字符，按 \n 分割
    ▼
完整的行（含 NNN| 前缀）
    │
    ▼
StreamingXmlParser.feed_line(line)
    │ 剥除 NNN| 前缀
    │ 正则匹配 XML 元素
    │ 更新状态机
    ▼
ParseEvent 事件流
    │ SEGMENT, CHOICE_BEGIN, OPT, BRIDGE, SET, CHECKPOINT, ...
    ▼
GameLoop.stream_round()
    │ 分发事件，校验状态
    ▼
UI（CLI / Web SSE）
```

StreamingXmlParser 的核心设计：
- **单遍扫描**：每次处理一行，立即产出事件，不缓存、不回看。
- **状态机跟踪**：维护 `_in_story`、`_in_branch`、`_in_checkpoint`、`_in_choice`、`_post_bridge` 等状态，确保嵌套关系正确。
- **NNN| 行号前缀**：每行输出有 `NNN|` 前缀（3 位补零递增序号）。解析器自动剥除，同时记录编号问题（跳号等）作为格式反馈。
- **格式错误检测**：bridge 后禁止出现 choice/set/checkpoint 等交互元素，违规则记录为格式错误反馈给下一轮 prompt。

### 3.5 双队列缓冲与多线程安全

> 前述三个理念（桥接、时序、流式解析）构成了无缝体验的理论框架。将它们落实到代码层面，需要解决一个具体的工程问题：**多个不同速率的生产者和消费者如何安全、高效地协作**。

#### 3.5.1 双队列缓冲

桥接机制和流式解析的落地，依赖**双队列缓冲**架构：

- **后台队列（`_pending_queue`）**：daemon 线程将 API 的 SSE token 片段推入 `queue.Queue`。
- **前台消费（`stream_round()`）**：主流程从队列中拉取 token，送入 LineBuffer → StreamingXmlParser → 事件流。
- **Queue 天然解耦**：生产和消费速度解耦——API 返回快慢不影响主流程的拉取节奏，主流程暂停等选择时 API 流继续在后台积累。

#### 3.5.2 Web UI 四层线程架构

Web UI 的线程模型是项目中多线程安全设计最复杂的部分，经历了 6 轮迭代修复才消除所有竞态条件：

```
[daemon 线程]                      API 流式调用
    │                                │
    ▼                                ▼
queue.Queue                      token 片段入队
    │
    ▼
[后台游戏线程 run_loop()]          从 Queue 取 token → stream_round()
    │                                │
    ▼                                ▼
queue.Queue (事件队列)             ParseEvent → dict → 入队
    │
    ▼
[async SSE generator]              轮询事件队列 (100ms) → JSON → SSE
    │
    ▼
[浏览器前端]                       EventSource → 事件队列 → 单消费者展示循环
```

关键并发安全保障：
- **Cancel Sentinel**：取消游戏时向活跃队列和待处理队列同时注入取消哨兵，覆盖"后台发起 API 调用"与"前台开始消费队列"之间的竞态窗口。
- **Identity-Checked 状态清理**：清理游戏流状态时使用身份比对（指针相等），防止旧的后台线程误删新 stream 的状态。
- **Local Stop-Event Reference**：SSE 端点使用局部变量捕获停止信号，防止并发创建新 stream 时意外覆盖。
- **三级解锁机制**：停止信号、选择等待唤醒、队列哨兵三者联动——确保阻塞中的线程能立即观察到退出信号。

### 3.6 本地数据优先原则

#### 3.6.1 核心理念

**所有游戏数据以本地 `GameState` 对象为唯一真实来源**。LLM 不能直接修改任何游戏状态——它只能通过 `<set>` 元素**建议**变更，程序对每项建议进行校验后决定是否接受。

#### 3.6.2 校验体系

引擎对每项状态变更建议执行 5 步校验：

1. **变量存在性**：目标变量名必须存在于共创阶段定义的变量列表中。
2. **操作合法性**：数值变量支持增减和赋值，标签变量仅支持赋值。
3. **值解析**：将 LLM 输出的文本值转换为正确的数据类型。
4. **条件评估**：`if` 属性中的条件表达式（支持"与""或"组合，最多一层），玩家本轮选择优先于全局状态变量。
5. **应用与钳位**：通过校验的数值变更自动钳位至 [0, 100]，标签变更直接赋值。

#### 3.6.3 设计哲学

本地数据优先原则在"AI 创造力"与"程序确定性"之间建立了清晰的边界：

- AI 负责**创意层面**——判断故事走向、建议状态变化。
- 程序负责**规范层面**——确保状态变化合法、一致、可预期。
- 被拒绝的变更**不报错中断游戏**，而是静默记录，在下一轮 prompt 中作为错误反馈告知 LLM，让其自行修正叙事方向。

### 3.7 错误处理理念

#### 3.7.1 两级分类

| 级别 | 类型 | 处理策略 |
|------|------|---------|
| **严重错误** | API 调用失败、流式超时、XML 解析失败 | 暂停游戏 → 通知 UI → 用户决定（重试/放弃） |
| **普通错误** | 格式问题（编号跳号、bridge 后交互元素）、状态变更被拒 | 静默处理 → 记录 → 下一轮 prompt 中反馈 LLM |

#### 3.7.2 不自动重试原则

所有 API 级别的失败**不自动重试**。原因：

- 自动重试掩盖了真实的 API 问题（如配额耗尽、认证失败），导致用户困惑。
- 每次 API 调用都消耗 token（=成本），擅自重试等于在用户不知情的情况下增加成本。
- 用户有权根据错误类型（超时可重试、配额用完应停止）做出最适合自己的决策。

程序的角色是**提供信息和选项，而非替用户做恢复决策**。

#### 3.7.3 Prompt 反馈机制

普通错误（格式问题、状态变更被拒）通过下一轮的 prompt 反馈给 LLM：

```
[Error Feedback]
- Rejected state changes: 勇气 += 200 (超出 0-100 范围，已钳位至 100)
- Format issues: Line 147 缺少 NNN| 编号前缀
```

这种"静默记录→下轮反馈"的机制避免了中断游戏体验，同时持续引导 LLM 输出更合规的内容。

### 3.8 国际化设计

#### 3.8.1 设计理念

国际化（i18n）从项目初期就已纳入设计考量，而非事后补充。基于以下理念：

- **引擎无国界**：核心引擎代码中不硬编码任何面向用户的语言字符串。
- **翻译可扩展**：添加新语言只需新增翻译文件，无需修改代码。
- **运行时切换**：玩家可在任意时刻切换语言，无需重启游戏。

#### 3.8.2 技术方案

- **后端**：Python 标准库 `gettext`，`.po/.mo` 编译流程。支持 zh-CN（简体中文）、zh-TW（繁体中文）、en（英文）。
- **前端**：构建时自动从 `.po` 文件生成 JavaScript i18n 字典（`i18n-dict.js`），前端无需额外网络请求。
- **语言元数据外部化**：LLM prompt 中的语言相关指令（中文的叙事风格指导、示例变量名等）存储在 `lang_meta/{lang}.json` 文件中。引擎读取当前语言的元数据注入 prompt。添加新语言 = 新增一个 JSON 文件 + 翻译 `.po` 文件，零代码改动。

### 3.9 引擎与 UI 分离

#### 3.9.1 设计原则

项目采用**双人协作模式**（引擎核心 + Web UI 独立开发），这要求引擎与 UI 之间有清晰的边界：

- **引擎核心（本人负责）**：提供能力，不关心展示形式。包括游戏循环、状态管理、LLM 交互、存档系统、解析器。
- **UI（合作者负责）**：独立决定展示方式、交互节奏、布局样式。包括终端 CLI + Web 界面。
- **连接层**：`GameSession`（统一入口）+ 生成器事件流（引擎通过 `yield` 逐事件推送，UI 通过 `send` 注入玩家选择）。

#### 3.9.2 文件所有权边界

引擎核心文件（UI 层只读不写）：
- `core/game_loop.py`、`core/co_create.py`、`core/context_manager.py`、`core/prompt_builder.py`、`core/save_manager.py`
- `parser/streaming_parser.py`
- `io/api_client.py`

UI 领地文件（引擎层不依赖）：
- `web/`（FastAPI + 静态前端）
- `locale/`（语言翻译和配置）

这一规则避免了双人并行开发中的合并冲突——两个开发者可以在各自领地内自由迭代，只需保持 `GameSession` API 兼容即可。

#### 3.9.3 生成器事件流

引擎与 UI 的核心通信机制是 Python 生成器：

```python
def stream_round(self) -> Iterator[dict]:
    yield {"type": "story_begin"}
    yield {"type": "segment", "text": "...", "branch": "main"}
    yield {"type": "bridge"}
    yield {"type": "options", "choices": [...], "opt_branches": {...}}
    key = yield  # 暂停，等待 UI 注入玩家选择
    yield {"type": "state", "variables": {...}}
    yield {"type": "done"}
```

UI 完全自主决定：
- 何时推送下一段文本（自动/手动模式）
- 选项如何布局（按钮排列、灰显样式）
- 状态面板展示哪些变量

引擎只负责"什么事件发生了"，UI 负责"怎么呈现"。这种基于生成器的暂停-恢复机制保证了引擎和 UI 可以独立演进。

## 4. 关键实践

> 第 3 章阐述了"我们设计了什么"。本章记录"我们如何确保设计被正确实现"——即从理念到落地的工程方法论。这些实践本身也是项目的重要产出：它们形成了一套可复用的 AI 辅助开发工作流。

### 4.1 规范文档驱动开发

项目确立了"以文档指导代码构建"的工程纪律：

- **所有设计决策先写入文档**：4 份权威 spec（`exec-flow.md`、`block-spec.md`、`prompt-design.md`、`data-model.md`）在代码编写前完成核心内容。
- **代码修改反馈回文档**：任何代码变更如果影响了设计语义，必须同步更新对应 spec。
- **任何问题以文档为最终仲裁**：当代码行为与 spec 描述不一致时，以 spec 为准——代码是 spec 的派生品，而非相反。

这套纪律在项目初期并不成熟——spec 和代码之间存在大量不一致。随着多次"spec-vs-code 审计"（07-11 发现 6 处、07-13 修复 16 项、07-16 修复 9 项），文档质量逐步提升，最终成为真正可信的权威参考。

### 4.2 提示词实验体系的构建

提示词迭代依赖两大基础设施：

1. **prompt_lab 工具集**（`tests/prompt_lab/`）：
   - `generate_prompt.py`：从 YAML 模板 + 参数矩阵生成测试用的 prompt 变体。
   - `run_prompt_test.py`：向 LLM 发送 prompt 并收集原始响应（含 TTFT）。
   - `analyze_frame.py`：分析响应——时序（TTFT） + XML 格式正确性（11 项检查）。
   - `analyze_seg_test.py`：针对 Segment 长度实验的专项分析。

2. **Dev CLI 观察者模式**：
   - 终端测试工具在每个叙事轮次完成后自动回调观察者，记录本轮完整数据。
   - 记录内容：实际发送给 LLM 的完整消息数组、原始响应文本、TTFT、token 用量、当前大纲节点、玩家所选分支。
   - 这使得每次测试运行都是可追溯的数据点，而非"跑完就忘"。

### 4.3 代码构建方法论的演变

回顾整个开发过程，代码构建的质量经历了三个阶段：

**阶段一（初期——效果远低于预期）**：
AI 被允许自行大量构建代码。由于初期设计文档不够精确、约束描述不够具体，AI 倾向于按照通用的编码惯性编写代码，产生了大量看似正确但实质偏离设计预期的实现。典型问题包括：以全量解析取代流式解析、忽略时序模型的约束、错误处理逻辑与设计原则矛盾。

**阶段二（中期——有所改善但仍不足）**：
引入 superpowers 技能包后，AI 的任务分解和执行质量有所提升，但仍存在"单次改动量过大导致细节丢失""忽略跨文件影响"等问题。

**阶段三（后期——严格规范）**：
确立了三条硬性规范，之后代码构建的准确性得到有效保障：

1. **逐步修改**：每次只改一个明确的点，一个对话只完成一个目标。
2. **先文档后代码**：任何修改先在 spec 中体现，再落实到代码。
3. **三要素说明**：每次修改必须说明（a）内容、（b）理由、（c）具体依据。

这个演变过程揭示了一个重要经验：**AI 辅助编程的效果不取决于 AI 的能力，而取决于人类给出的约束质量**。初期约束粗糙 → 结果粗糙；后期约束精确 → 结果精确。

### 4.4 Dev CLI——被低估的基础设施

Dev CLI 看似"落后"（终端界面），实则在项目开发中发挥了不可替代的作用：

- **提示词调试**：观察者模式截取并记录实际发送给 LLM 的完整 messages 数组——这是 Web UI 无法提供的底层可见性。
- **TTFT 监控**：每轮自动记录 TTFT，为提示词优化提供数据支撑。
- **格式检查**：DevObserver 在解析完成后自动运行格式规范检查，快速发现 LLM 输出的格式问题。
- **Web UI 参考实现**：Web UI 的叙事循环调用模式、事件分发逻辑，均以 Dev CLI 为蓝本。

### 4.5 线程安全——迭代次数最多的工程挑战

Web UI 的线程安全问题共经历 6 轮迭代才完全消除竞态条件。其根本困难不在于单次设计的复杂度，而在于**并发 bug 的暴露高度依赖运行时时序**——同一份代码在大多数运行中表现正常，仅在特定时序交错时崩溃。

问题收敛过程揭示了并发调试的一个典型模式：

- **第 1-2 轮**：修复明显的数据竞争（game stream 的全局状态被并发覆盖）。
- **第 3-4 轮**：修复退出路径的竞态——用户关闭浏览器标签时，daemon 线程、游戏线程、asyncio 协程三者可能同时调用清理逻辑。
- **第 5-6 轮**：修复重入保护——快速连续点击"开始游戏"会创建多个 stream，旧 stream 的 daemon 线程可能在新 stream 启动后才完成清理，误删新 stream 的状态。

解决方案的核心机制（cancel sentinel、identity-checked 状态清理、local stop-event 引用捕获、三级解锁）已在 3.5.2 中详述。这里强调的是方法论的教训：**并发安全不能靠"看起来没问题"来判断，必须枚举所有可能的线程交错路径并逐一设防**。

## 5. 课程内容联系

> 前文的设计理念和工程实践，在底层落实为具体的程序逻辑和数据结构。本章回到课程视角，系统梳理项目与《程序设计》《数据结构》知识点的对应关系。"⭐"标记为深度应用——这些知识点不是被简单地"用到"，而是与核心设计理念形成结构性依赖。

### 5.1 《程序设计》

#### I/O 流（⭐ 核心亮点）

"流"的概念贯穿项目设计的各个层面，是最能体现课程核心概念的知识点：

| 层次 | 实现 | 对应课程概念 |
|------|------|------------|
| **网络流** | HTTP SSE（Server-Sent Events）流式接收 LLM 输出 | 字节流、缓冲读取、流式解析 |
| **文件流** | 存档的原子写入（temp file → `os.replace`） | 文件 I/O、原子操作、数据持久化 |
| **解析流** | LineBuffer 累积字符 → 按 `\n` 分割 → StreamingXmlParser 逐行消费 | 字符流、行缓冲、流水线处理 |

这三层流不是孤立的技术点——它们被统一在时序模型框架下协同工作：网络流的速度决定了上游输入速率，解析流的逐行处理使得首行即可展示，文件流保证状态可靠落盘。**"流"既是一种技术实现，也是一种设计哲学**——数据不应该被"囤积"再"批量处理"，而应该在产生的同时就被消费和呈现。

#### 异常处理

项目实现了分层的异常处理体系：

- **通信层**：捕获 HTTP 错误，解析服务端返回的错误信息，封装为统一异常类型向上传递。
- **引擎层**：区分可恢复错误（暂停并通知 UI，提供重试入口）与不可恢复错误（终止当前流程）。
- **UI 层**：将引擎异常映射为用户可读的错误提示，由用户决定下一步操作。

重要设计决策：**不自动重试 API 调用**。体现了"异常处理不是自动隐藏错误，而是正确分类并交由合适的决策层处理"的理念。

#### 字符串处理

- **XML 逐行解析**：正则表达式匹配各类 XML 标签，属性提取（`summary="..."`、`branch="..."`），文本内容提取（`<seg>` 标签内文本）。
- **i18n 字符串替换**：`gettext` 的 `_()` 函数贯穿 UI 代码，支持占位符和复数形式。
- **行号前缀剥除**：`NNN|` 前缀的正则匹配和剥除，同时检测编号连续性。

#### 函数与模块化

项目按职责划分为清晰的分层结构：

```
storyloom/
├── core/       # 游戏引擎核心（game_loop, context_manager, co_create, save_manager, session）
├── parser/     # 解析器（streaming_parser）
├── io/         # API 客户端（api_client）
├── web/        # Web UI（server, sessions, static/）
└── dev_cli/    # 开发者终端
```

模块间的依赖关系是单向的、可追溯的：游戏循环依赖提示词构建器生成消息、依赖上下文管理器维护对话历史、依赖 API 客户端发起网络请求、依赖流式解析器处理响应。每个模块只知道自己向下的依赖，不感知上层调用者。

#### 循环控制

初期使用无限循环和递归导致 WSL 资源耗尽（递归深度过大）。后改为**有限循环 + 迭代计数器**：后台线程有限循环消费 SSE 流，叙事主循环有限轮次迭代。这一经历加深了对循环边界条件重要性的理解。

#### 面向对象

核心组件均封装为类，通过对象组合构建系统：

| 类 | 职责 | 关键方法 |
|---|------|---------|
| `GameState` | 游戏状态管理、变更校验 | `apply_set()`, `evaluate_condition()` |
| `ContextManager` | 对话历史、滑动窗口、压缩 | `set_round1()`, `add_round()`, `get_messages()` |
| `StreamingXmlParser` | 流式 XML 解析 | `feed_line()`, `get_result()` |
| `GameLoop` | 游戏循环编排 | `start_game()`, `stream_round()`, `retry()` |
| `ApiClient` | LLM API 通信 | `chat()`, `stream_chat_iter()` |
| `SaveManager` | 原子化存档 | `save()`, `load()`, `list_saves()` |
| `CoCreateFlow` | 共创流程状态机 | `start()`, `send()`, `generate()` |

#### 列表与字典

- **消息数组**（`list[dict]`）：对话历史由有序的 `{role, content}` 字典列表组成，是 ContextManager 的核心数据结构。
- **状态变量**（`dict[str, int|str]`）：游戏运行时状态以字典存储，支持 O(1) 读写。
- **选项字典**（`dict[str, str]`）：玩家选择 → 分支名称的映射。
- **大纲节点列表**（`list[dict]`）：有序节点序列，每个节点含 routes、status、summary 等字段。

### 5.2 《数据结构》

#### 队列（⭐ 深度应用，与桥接机制直接关联）

`queue.Queue` 是项目中最重要的数据结构之一，出现在两个关键位置：

**Bridge 预取队列**：
```python
# daemon 线程：API 返回的 token 逐个入队
_queue.put({"delta": token})

# 主线程：stream_round() 逐个出队消费
chunk = _pending_queue.get()
```

这是桥接机制能够运转的物理基础——daemon 线程和主线程通过 Queue 解耦，生产速度（API 返回）与消费速度（解析展示）完全独立。

**Web UI 事件管道**：
```
后台游戏线程 → Queue[ParseEvent] → 异步 SSE generator → 浏览器
```

浏览器端的 JavaScript 也实现了相同模式——SSE 事件进入 `_eventQueue`（数组模拟队列），单消费者展示循环按固定节奏出队。

队列在这个项目中的应用不是简单的"用了"，而是**深度嵌入核心设计——没有队列就没有桥接机制，没有桥接就没有无缝体验**。

#### 滑动窗口（⭐ 深度应用，与提示词设计直接关联）

`ContextManager` 维护对话历史的固定大小窗口：

```
[0] Round 1 user  ───────────── 永久锚定 ─────────────
[1] Round 1 assistant ───────── 永不删除 ─────────────
[...] 压缩轮次摘要 ──────────── 单条 user/assistant 对 ──
[...] 最近 WINDOW_SIZE 轮 ──── 完整保留 ─────────────
[last] 当前轮 user ──────────── 待 LLM 响应 ────────────
```

核心参数：
- `WINDOW_SIZE = 3`：始终保持 3 轮完整上下文。
- `FIRST_COMPRESSION_AT = 5`：第 5 轮起触发首次压缩。
- `MAX_CONTEXT_TOKENS ≈ 50,000`：目标上下文预算。

这是一个典型的**固定大小滑动窗口 + 摘要压缩**问题，与 TCP 拥塞控制中的滑动窗口、数据库查询中的游标分页有相似的数据结构本质。

#### 有向图（故事大纲建模）

故事大纲被建模为 checkpoint 节点 + route 边的有向图：

```
[ch1_intro] ──(无条件)──▶ [ch2_rising] ──(if 勇气>50)──▶ [ch3_heroic]
                                        ──(else)──────▶ [ch3_cautious]
```

Route 条件评估规则：
1. 按顺序评估所有 route 的 `if` 条件。
2. 第一个条件满足的 route 胜出。
3. 如果所有条件都不满足 → 回退到第一条无条件 route。
4. 如果不存在任何 route → 当前节点为结局节点。

游戏的大纲在共创阶段由 LLM 生成，在叙事阶段由引擎根据 checkpoint 事件驱动推进。玩家选择通过状态变量间接影响路由方向。

#### 栈（XML 标签嵌套匹配）

StreamingXmlParser 使用状态栈跟踪 XML 元素嵌套：

```python
# <choice> 开标签 → 进入 choice 上下文
_in_choice = True

# <opt ...> → 在 choice 上下文中解析选项

# </choice> 闭标签 → 退出 choice 上下文
_in_choice = False
```

嵌套关系保证了 `<branch>` 内的 `<seg>` 被正确归属、`<choice>` 内的 `<opt>` 不泄露到外部。

#### 树形结构（XML 解析结果）

解析完成后的数据结构形成嵌套树：

```
ParsedOutput
├── segments: list[Segment]        # 全部叙事段
│   ├── Segment (pre, main)        # bridge 前的段
│   └── Segment (post, branch_a)   # bridge 后的段（按 branch 分类）
├── choices: list[dict]            # 选项列表
│   ├── {"key": "1", "text": "...", "branch": "attack"}
│   └── {"key": "2", "text": "...", "branch": "negotiate"}
├── sets: list[SetOperation]       # 状态变更建议
├── checkpoint_node: str           # 路由节点
└── routes: list[RouteTarget]      # 出边路由
```

这种多层嵌套结构通过 dataclass 实现，便于序列化（存档）和递归遍历（文本收集）。

#### 哈希表（O(1) 状态查找）

- **状态变量**：`state_vars: dict[str, int|str]`，条件评估中 O(1) 取值。
- **选项字典**：`choice_dict: dict[str, str]`，选项 key → 分支名称映射。
- **条件变量优先级**：解析条件表达式时，先在 `choice_dict` 中查找，未命中再查 `state_vars`——两层哈希查找均 O(1)。
- **分支文本过滤**：桥接后文本按分支名称归类存储，根据当前所选分支以 O(1) 取出对应文本。

## 6. 主要工作内容

> 前面的章节展示了"做了什么"和"为什么这样做"。本章转向"谁做了什么"——明确界定人类与 AI 在项目中的工作边界。在一个深度使用 AI 辅助开发的项目中，这种界定既是学术诚信的要求，也是理解人机协作模式的关键。

### 6.1 人类工作（项目负责人——本人）

#### 理念与设计（全部）

- 项目全部设计理念的**初步构想和方向确立**：AI 实时生成式游戏的核心概念、桥接机制的灵感、时序模型的总结、流式解析的坚持。
- 项目全部**关键设计决策**：XML 输出格式的选择、NNN| 行号前缀的引入、对话式消息数组架构、bridge 位置 (75%)、本地数据优先原则、不自动重试原则。

## 7. 构建流程

> 设计理念和工程实践并非一次性确立的——它们是在 20 天的密集开发中逐步浮现、验证和完善的。本章按时间线梳理构建过程，重点展示关键决策的形成时机和背景。
> <!-- UPDATE: 后续阶段追加到时间线末尾 -->

### 7.1 阶段一：构想与设计奠基（07-02 至 07-03）

- 确立项目核心概念：桥接机制、两层分支（段内/大纲）、本地数据优先。

### 7.2 阶段二：提示词实验与格式迭代（07-04 至 07-05）

### 7.6 关键转折点：方法论的演变

整个构建流程中最关键的转折点发生在阶段三前后——从 **"AI 大量自行构建"** 到 **"逐步修改 + 先文档后代码 + 三要素"** 的严格规范。

初期由于设计文档不够精确，AI 倾向于大量生成"看似正确"的代码，但大量细节与设计理念不符（全量解析替代流式解析、忽略时序模型、错误处理与设计矛盾等）。这段经历虽然造成了返工，但也揭示了 AI 辅助开发的核心规律：**AI 的输出质量由人类约束的精确度决定**。此后确立的三条硬性规范成为项目后期高效推进的保障。

## 8. AI 使用感悟

> 本章是前述内容的自然延伸——第 6 章界定了人类与 AI 的工作边界，第 7 章记录了方法论如何在实践中演变。本章将这些经验提炼为可推广的原则，旨在为未来的 AI 辅助开发提供参考框架。

### 8.1 确保自身理解深入——LLM 在"记忆"上远不如人

无论是项目构建过程中对 LLM 的约束，还是游戏运行时对 LLM 的 prompt 约束，都反复验证了一个规律：**LLM 有极强的惯性思维**。许多重要要求总在多次强调后被忽略——例如序号的惯性延续（关掉编号后 LLM 仍然自动加编号）、正面要求和反面禁止同时存在的情况下仍然出错。

这意味着：如果人类自己对项目的设计理解不够深入，就无法识别 LLM 输出中"看起来合理但实质偏离"的部分。**LLM 是强大的执行者，但不是可靠的设计者——设计深度的底线由人类把控**。

### 8.2 先改文档后改代码

"只改代码"总会造成文档过时——这一教训在项目中被反复验证。后来确立了铁律：**任何修改必须先在文档中体现，再进行代码实现**。这不仅保证了文档的持续准确性，更强迫每次修改前进行"设计层面的推敲"——文档写不清楚的地方，代码也不可能写对。

### 8.3 三要素：内容、理由、依据

约束 LLM 行为的最有效方式是要求其在每次修改时说明三要素：

1. **内容**：具体改了什么。
2. **理由**：为什么这样改。
3. **依据**：依据哪份文档的哪个条款。

这既规范了 LLM 的行为（减少"擅自发挥"），也让人类 review 者能快速判断修改的合理性。三要素规则同时保证了"方向正确"（有文档依据 = 不偏离设计）和"可审查性"（有理由 = 可判断修改是否适当）。

### 8.4 每次只完成一个小工作

一个对话只应当完成一个目标。让 LLM 在一次对话中做大量工作时，通常会：
- 忽略诸多细节。
- 不按规范设计（倾向于"快速完成"而牺牲精度）。
- 不好人为控制（修改量大导致 review 困难）。

小步快跑的策略在后期被严格执行：每次改动范围可控，review 质量有保障，发现问题能快速定位。

### 8.5 审查、明白每一步所做的工作

不要无脑同意 AI 的修改。项目构建过程中曾无数次发现 AI 设计和修改的问题，全部被纠正。

**其中最重要的一次：流式解析设计的重要辩论**。AI 在代码重构中删除了 StreamingXmlParser，认为全量解析"更简单"。我审查后坚持恢复流式解析——判断依据不是代码层面的便利性，而是时序模型揭示的用户体验本质（TTFT vs 总生成时间）。事后证明流式解析是完全正确且必要的设计。**这次经历说明：人类对设计原则的理解深度，是 AI 无法替代的最后一道防线**。

### 8.6 可推广性

以上五条感悟不限于本项目。它们构成了一套可推广的 **AI 协作方法论**：

1. **人类负责设计深度**（理解→决策→把关）。
2. **文档是协作的契约**（先写清楚再动手）。
3. **三要素是质量保障**（内容+理由+依据 = 可审查的最小单元）。
4. **小步快跑**（范围可控→质量可控）。
5. **每步必审**（不盲信、不跳步）。

这套方法论将应用于未来的 vibe-coding 实践——不是"让 AI 自由发挥"，而是"人类设定精确约束、AI 在约束内高效执行、人类逐步审查验收"。
