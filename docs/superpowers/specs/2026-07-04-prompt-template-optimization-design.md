## 一、背景与动机

Phase 1 Prompt 设计的核心要求是双重的：

1. **正确性**：LLM 返回的剧情段格式正确（区块分隔符、分支路由、state 语法、checkpoint 节点引用）
2. **无缝性**：bridge 机制确保用户不感知段边界（下一轮首段在尾部缓冲播完前到达）

默认 Prompt（3329 chars）在 5 次测试中正确率仅 ~33%，TTFT 平均 56s。深入分析后发现：

- **正确性问题的根因**：Prompt 依赖 LLM 从格式示例"推断"规则，而非显式定义禁止边界
- **无缝性问题的根因**：原 BRIDGE_SEGMENT_RATIO=0.75 导致尾部缓冲仅 5s，而首段到达需要 38s+
- **对 bridge 机制的理解偏差**：早期测试使用 `stream=False`，误将总生成时间当作时限。实际上流式模式下只需首段在缓冲播完前到达

## 二、方法论

### 2.1 测试基础设施

在优化 Prompt 之前先修复了测试工具：

| 修复 | 说明 |
|------|------|
| 流式串行默认 | `stream=True` + 串行执行，测量真实 TTFT 和 FirstSegment |
| 并行 → 串行 | 发现并行导致 TTFT 翻倍（服务端排队），改为默认串行 |
| 正确性自动化 | `analyze_results.py` 支持一键运行 8 项正确性检查 + 时序分析 |

### 2.2 迭代流程

```
跑 3 次测试 → 检查正确性 + 无缝性 → 定位根因 →
应用设计原则修改 Prompt → 对比测试 → 记录
```

共 6 轮迭代、30+ 次测试，测试了 8 个 Prompt 变体。

## 三、核心发现：七条通用设计原则

经过对 default → v2-lean → v2 → v2-final → v2-detailed → v3 → v4 的逐个对比，提炼出以下原则。它们的有效性通过正确率（33%→83%）和 TTFT（38s→11s）的量化改善得到验证。

### 原则 1：反例约束

**只说"不能做什么"不够——给出具体会被拒绝的写法。**

| 写法 | 效果 |
|------|------|
| `禁止修改 node_id` | ❌ LLM 不理解"修改"的边界 |
| `禁止 ch2_confrontation_resolved（拼接后缀）` | ✅ LLM 看到具体反例，直接避开 |

应用场景：checkpoint node、bridge 分支名、choice 变量名。

### 原则 2：正反双重覆盖

**关键约束在正面规则和负面禁止中各出现一次。** 单次提及的被漏看概率 ~30%，双重覆盖后接近 0%。

```
正面: bridge 之前必须使用 `:main` 分支
负面（禁止节）: bridge 之前使用非 `:main` 分支 ← 禁止
```

### 原则 3：注意力标签

**用 `（重要）` 标记最易出错的规则节。** v2 的 checkpoint 没标标签 → 2/3 正确。v3 标了 → 6/6 正确。标签告诉 LLM 往哪分配有限的注意力资源。

### 原则 4：示例-规则屏障

**格式示例结束后加显式提醒，防止 LLM 将示例续写为自己的输出。**

```
（以上为格式示例。你的输出是全新的剧情段，必须从 1 开始编号，
 以 --- narrative:main --- 开头。）
```

v3 测试中 1/6 的"从示例编号 15 续写"故障在 v4 加屏障后消除。

### 原则 5：具体优于抽象

**给出数字和案例，而非比例或泛化描述。** LLM 对"40%"的计算不可靠，但看到"32 段后"就能执行。

```
总 80 段 → bridge 在第 32 段后 ✓
总 100 段 → bridge 在第 40 段后 ✓
禁止 bridge 之后尾部不足 20 段 ✗
```

### 原则 6：显式禁止优于隐式模式

**不要依赖示例教会 LLM"不能做什么"。** 示例展示正确格式，规则定义禁止边界。独立的 `**禁止**` 节逐条列出禁止行为——每条都是测试中实际出现过的错误。

### 原则 7：关键处不吝笔墨

**整体追求紧凑，但在反复出错的规则上多花 tokens。** checkpoint 和 options 的正确率从 33%→100% 靠的是规则更详细，不是更短。v4 比 default 还少 49 chars，但关键规则更详尽。

## 四、最终方案：v4 Prompt 模板

### 4.1 结构

```
角色定义（1 行）
  ↓
格式示例（11 段 pre-bridge + 5 段 tail）
  ↓
示例-规则屏障（1 行提醒）
  ↓
核心规则（7 节，按优先级排列）
  ├── 段格式（重要）
  ├── 编号
  ├── options（重要）
  ├── 段数与 bridge（重要）
  ├── state
  ├── checkpoint（重要）
  └── 禁止
  ↓
质量要求
  ↓
故事上下文（占位符替换）
```

### 4.2 关键参数

| 参数 | 旧值 | 新值 | 依据 |
|------|------|------|------|
| SEGMENTS_PER_ROUND_MIN | 30 | 60 | 需要更多 tail 缓冲 |
| SEGMENTS_PER_ROUND_MAX | 50 | 120 | 匹配优化后的 TTFT |
| BRIDGE_SEGMENT_RATIO | 0.75 | 0.4 | bridge 提前，增加缓冲 |

### 4.3 量化成果

| 指标 | default | v4 | 改善 |
|------|---------|-----|------|
| 系统 Prompt 大小 | 3329 chars | 3280 chars | -1.5% |
| TTFT 平均 | 38s | 11s | **3.5×** |
| 正确率 | 33% (1/3) | 83% (5/6) | **2.5×** |
| 无缝率 | 33% (1/3) | 83% (5/6) | **2.5×** |
| choice 缺失 | 偶发 | 0 | 消除 |
| pre-bridge 分支错误 | 偶发 | 0 | 消除 |
| checkpoint node 虚构 | 67% | 0 | 消除 |
| 编号从 1 开始 | 偶发 | 5/6 | 基本消除 |

## 五、适用范围

七条原则是跨场景通用的——不依赖特定故事题材、变量系统或大纲结构。任何需要 LLM 输出结构化内容的 Prompt 设计都可以应用：

1. **新故事场景**：替换 `# 故事` 上下文即可，格式和规则部分不变
2. **不同 LLM**：原则是模型无关的。如果切换到更快/更慢的模型，只需调整段数和 RATIO 参数
3. **扩展区块类型**：新增区块类型时，遵循"示例 + 反例 + 双重覆盖"模式添加规则

## 六、遗留与展望

- **段数上限控制**：LLM 偶尔超出 120 段（Test 01 的 121 段）。当前偏差仅 1 段，可接受。后续可考虑更严格的措辞
- **编号从 1 开始**：v3 有 1/6 的"示例续写"故障，v4 加屏障后未复现。需更多测试确认长期稳定性
- **不同模型适配**：当前测试基于 deepseek-v4-pro。切换到其他模型时需重新校准 TTFT 和段数参数
