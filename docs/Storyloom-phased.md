# Storyloom: AI 交互式文字小说游戏引擎（分阶段实施）

> **原完整需求文档**：`docs/Storyloom.md`（保留作为远期参考）
> **当前文档**：将完整愿景拆分为三个可独立交付的阶段，Phase 1 为重点。

---

## 项目总览

**一句话定位**：以 LLM 为叙事大脑、程序为流程管理器的 AI 交互式文字冒险游戏引擎。

```
Phase 1 ──── CLI 纯文本 MVP ──── 单模型、硬编码状态、线性叙事 ──── 🎯 当前
Phase 2 ──── Web UI + 动态系统 ── 动态状态、向量记忆、多模型
Phase 3 ──── 完整体验 ────────── 图像生成、云同步、TTS、导出
```

---

## Phase 1: 纯文本单模型 MVP

> **核心原则**：纯文本、单一 API key、不做 token 预算控制、最快跑通完整游戏循环。

### 1.1 目标与范围

#### Phase 1 是什么

- 一个 **终端 CLI** 文字冒险游戏，用户通过键盘选择选项推进故事
- 使用 **单一 OpenAI 兼容 API**（一个 key 覆盖所有 LLM 调用）
- **三套硬编码状态模板**（恋爱/冒险/悬疑），用户选题材即载入
- **线性叙事 + 可选关键分支**：大纲为节点树，每轮推动剧情至下一节点
- **简单记忆**：保留最近 N 轮原文 + LLM 定期生成摘要
- **本地 JSON 存档**，支持多槽位

#### Phase 1 不是什么

- ❌ 没有图像生成（Stable Diffusion / DALL·E）
- ❌ 没有 Web 界面（纯终端）
- ❌ 没有向量数据库 / embedding 检索
- ❌ 没有多模型调度（叙事、审查、摘要用同一模型）
- ❌ 没有自定义文本输入（仅固定选项，§6.2 留到 Phase 2）

- ❌ 不做主动 token 计数与预算控制

### 1.2 技术选型

| 层面 | 选择 | 说明 |
|------|------|------|
| 语言 | Python 3.10+ | 标准库丰富，生态成熟 |

**依赖清单** (`requirements.txt`)：
```

```

### 1.3 核心架构

单进程事件驱动，模块划分如下：

```
用户输入 → UI (rich) → GameLoop → PromptBuilder → APIClient (stream) → Parser → GameState → 循环
```

| 模块 | 职责 |
|------|------|
| `main.py` | 入口：主菜单、新建/加载/设置 |
| `config.py` | 读取 `.env`，管理模型参数 |
| `api_client.py` | 封装 OpenAI SDK，支持流式输出 |
| `game_state.py` | 游戏状态数据模型 + 存档读写 |
| `prompt_builder.py` | 根据阶段组装 System/User Prompt |
| `parser.py` | 解析 LLM 返回的 XML 结构化输出 |
| `game_loop.py` | 共创阶段 + 叙事循环 + 结局处理 |
| `ui.py` | `rich` 终端界面：逐句展示、选项面板、侧边栏 |

### 1.4 游戏流程（简化版）

```
┌─────────────────────────────────────────────────────┐
│                    主菜单                            │
│         [1] 新游戏  [2] 继续  [3] 设置               │
└─────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│  阶段一：共创（设定 & 大纲）                          │
│                                                      │
│  1. 用户输入一句话想法（"想玩一个赛博朋克爱情故事"）    │
│  2. AI 追问 3-5 轮，澄清题材、世界观、主角             │
│  3. 用户选择题材模板 → 加载对应状态变量                │
│  4. AI 生成结构化故事设定（JSON）                      │
│  5. 用户确认设定                                      │
│  6. AI 生成大纲树（节点 + 目标 + 分支条件）            │
│  7. 直接进入游戏（大纲不要求确认）                     │
└─────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│  阶段二：叙事循环                                     │
│                                                      │
│  循环 {                                              │
│    1. PromptBuilder 组装上下文                        │
│    2. APIClient 流式请求 LLM                         │
│    3. Parser 解析 XML 结构化输出                       │
│    4. UI 逐句展示 <narration>（按 LLM 标记的分句边界）   │
│    5. UI 展示选项面板                                  │
│    6. 玩家选择选项 (1-5)                              │
│    7. 应用 <state_changes> 到 GameState               │
│    8. 检查 <checkpoint> 推进大纲节点                   │
│    9. 若触发结局 → 跳出循环                           │
│   10. 若轮次达阈值 → 触发摘要压缩                      │
│  }                                                    │
└─────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│  阶段三：结局 & 回顾                                  │
│                                                      │
│  1. AI 生成结局段落                                   │
│  2. 生成冒险日志（章节摘要 + 关键决策 + 状态曲线）     │
│  3. 展示回档选项（所有关键节点）                       │
│  4. 返回主菜单                                        │
└─────────────────────────────────────────────────────┘
```

### 1.5 状态系统（硬编码模板）

Phase 1 不要求 LLM 动态定义状态变量，而是预设三套模板。用户在共创阶段选择题材后自动载入。

#### 模板 A：恋爱

```json
{
  "genre": "romance",
  "vars": {
    "好感度": {"value": 0, "type": "int", "min": -100, "max": 100},
    "信任度": {"value": 0, "type": "int", "min": 0, "max": 100},
    "当前情绪": {"value": "平静", "type": "enum", "options": ["平静", "喜悦", "悲伤", "愤怒", "紧张"]},
    "关系阶段": {"value": "初识", "type": "enum", "options": ["初识", "朋友", "暧昧", "恋人", "疏远"]}
  }
}
```

#### 模板 B：冒险

```json
{
  "genre": "adventure",
  "vars": {
    "体力": {"value": 100, "type": "int", "min": 0, "max": 100},
    "金币": {"value": 500, "type": "int", "min": 0, "max": 999999},
    "声望": {"value": 0, "type": "int", "min": -100, "max": 100},
    "背包": {"value": [], "type": "list"},
    "队伍": {"value": [], "type": "list"}
  }
}
```

#### 模板 C：悬疑

```json
{
  "genre": "mystery",
  "vars": {
    "理智值": {"value": 100, "type": "int", "min": 0, "max": 100},
    "线索": {"value": [], "type": "list"},
    "时间进度": {"value": "第1天 上午", "type": "string"},
    "嫌疑度": {"value": 0, "type": "int", "min": 0, "max": 100}
  }
}
```

#### 状态更新规则

- LLM 在 `<state_changes>` 中声明变更，程序**严格校验**：
  - 变量名必须存在于当前模板
  - `int` 类型不能超出 `min`/`max`
  - `enum` 类型的值必须在 `options` 列表中
  - `list` 类型支持 `add`（追加元素）和 `remove`（移除元素）
- 校验失败 → 丢弃该条变更，记日志，继续执行（不中断游戏）

#### 状态注入格式

每轮 Prompt 中注入当前状态摘要，格式如下：

```
当前状态：好感度 30 | 信任度 45 | 情绪 平静 | 关系阶段 朋友
```

### 1.6 上下文管理（简化版）

Phase 1 不做向量检索，使用 **滑动窗口 + LLM 摘要** 两层结构：

```
┌────────────────────────────────────────────┐
│  System Prompt（固定部分）                   │
│  - 游戏规则                                  │
│  - 故事设定（JSON）                          │
│  - 大纲树当前章节目标                         │
│  - 长期摘要（如有）                           │
│  - 当前状态快照                              │
├────────────────────────────────────────────┤
│  User Messages（动态部分）                    │
│  - 最近 N 轮对话原文（N 默认 5）              │
│  - 早于 N 轮的内容 → 已被摘要替代             │
└────────────────────────────────────────────┘
```

#### 摘要触发条件

- 当对话轮次超过 `N * 2`（即缓存了 10 轮原文），触发摘要
- 将最早的 N 轮发送给 LLM，要求生成 **200 字以内** 的摘要
- 摘要存入 `GameState.memory.summary`，原文丢弃
- 后续每新增 N 轮，LLM 将 **已有摘要 + 新 N 轮** 合并成新摘要
- **注意**：摘要生成期间的游戏等待体验问题推迟到 Phase 2 处理。Phase 1 中摘要生成与叙事请求共用同一模型，同步等待。

#### 摘要 Prompt 要点

```
请将以下游戏剧情片段压缩为不超过 200 字的摘要，
保留关键事件、人物互动和状态变化，去掉对话细节。
```

### 1.7 结构化输出规范

所有 LLM 叙事响应必须遵循严格的 XML 格式。

#### XML Schema

```xml
<narration>
这里是故事叙述的正文段落……
可以包含多个自然段。
在合适的位置，程序会逐句展示。
</narration>

<options>
- 选项一的描述 (条件: 好感度>=20)
- 选项二的描述
- 选项三的描述
- 选项四的描述
</options>

<state_changes>
[
  {"var": "好感度", "op": "add", "value": 5},
  {"var": "金币", "op": "sub", "value": 100},
  {"var": "背包", "op": "add", "value": "旧钥匙"},
  {"var": "当前情绪", "op": "set", "value": "喜悦"}
]
</state_changes>

<checkpoint type="major" node_id="ch2_discovery" />
```

#### 标签说明

| 标签 | 必需 | 说明 |
|------|------|------|
| `<narration>` | ✅ 必选 | 正文叙述，支持多段落，长度由 LLM 自行控制 |
| `<options>` | 可选 | 2-5 个选项，每行以 `- ` 开头；可附带条件 `(条件: 变量>=值)` |
| `<state_changes>` | 可选 | JSON 数组，操作类型：`add`(+数值)/`sub`(-数值)/`set`(设值)/`add`(list追加)/`remove`(list移除) |
| `<checkpoint>` | 可选 | `type="major"` 关键节点（存档点），`type="ending"` 结局触发 |

#### 解析流程

```
原始响应
  → 正则提取 <narration>...</narration>  → 纯文本正文
  → 正则提取 <options>...</options>       → 解析选项列表 + 条件
  → 正则提取 <state_changes>...</state_changes> → JSON.parse → 校验
  → 正则提取 <checkpoint ... />           → 节点信息
```

#### 解析失败处理

1. 若 `<narration>` 缺失 → **重试**（附带错误提示："必须包含 <narration> 标签"），最多重试 2 次
2. 若两次重试均失败：
   - **有上一次有效响应** → 使用上一次有效响应作为回退
   - **无上一次有效响应（首轮/共创阶段）** → 展示错误信息，**询问玩家是否重试**（玩家选择重试则重新请求 LLM，选择放弃则返回主菜单）
3. 若 `<options>` 解析失败 → 静默丢弃选项（记日志），仅展示正文，**自动推进下一轮**（不提示玩家按键，不临时请求 LLM）
4. 若 `<state_changes>` JSON 解析失败 → 丢弃变更，记日志，继续执行
5. 若 `<options>` 中个别条目的条件解析失败 → **严格丢弃**该条目（不展示），只展示解析成功的选项

### 1.9 关键 Prompt 模板

Phase 1 所有 Prompt 使用中文。

#### A. 系统提示（System Prompt）

```
你是一个交互式文字冒险游戏的叙事引擎。你必须严格按照以下规则运作：

## 故事背景
- 题材：{genre_label}
- 世界观：{world_setting}
- 主角：{protagonist_name}，{protagonist_trait}
- 叙事风格：{tone}

## 当前章节目标
{current_node_goal}

## 当前游戏状态
{state_summary}

## 历史摘要
{memory_summary}

## 输出格式要求
你必须用以下 XML 格式回复，不要输出任何 XML 标签之外的内容：

<narration>
...正文叙述（自然段落，长度适中，在关键剧情点自然收尾）...

程序会以自然段（空行分隔）为单位逐段展示。请在需要停顿的位置用空行分隔。
</narration>

<options>
- 选项1描述
- 选项2描述
</options>

<state_changes>
[...]
</state_changes>

<checkpoint type="major" node_id="..." />

## 完整格式示例
以下是一个完整的正确输出示例：

<narration>
夜幕降临，新东京的霓虹灯在雨雾中晕开一片迷离的光。

你推开地下酒吧的门，潮湿的空气混杂着合成酒精和烧焦电路的气味扑面而来。吧台深处的角落里，一个戴着全息面具的人朝你招了招手。
</narration>

<options>
- 径直走过去，开门见山谈正事
- 先在吧台点一杯酒，观察周围环境 (条件: 理智值>=50)
- 假装没看见，转身离开
</options>

<state_changes>
[
  {"var": "理智值", "op": "sub", "value": 5},
  {"var": "线索", "op": "add", "value": "神秘人的全息面具"}
]
</state_changes>

规则：
1. <narration> 必须存在，控制在 200-500 字；用空行分隔展示单元，程序按空行逐段展示
2. <options> 每轮提供 2-4 个选项，选项之间要有信息差和不同的可能后果
3. 选项可以附带条件，格式：(条件: 变量>=值)，例如 "- 表白 (条件: 好感度>=50)"
4. <state_changes> 仅在剧情确实导致状态变化时出现
5. <checkpoint> 仅在到达大纲标注的关键节点时出现
6. 叙事必须与当前状态数值自然结合（如好感度低时角色更冷淡）
7. 若希望结束本章/游戏，使用 type="ending" 的 checkpoint
```

#### B. 共创阶段：追问 Prompt

```
你正在帮助玩家创建一个文字冒险游戏的故事设定。

玩家说："{user_input}"

请针对这个想法提出 3-5 个关键问题，帮助完善：
- 世界观的具体设定
- 主角的性格与背景
- 故事的基调与风格
- 主要冲突方向

不要直接给出建议，而是以提问方式引导玩家思考。
你的回复应该是一系列问题，每个问题附带 2-3 个选项供玩家选择或启发思考。
```

#### C. 共创阶段：生成故事设定

```
根据以下讨论内容，生成结构化的故事设定 JSON：

讨论记录：
{co_creation_history}

请输出如下格式的 JSON（仅 JSON，不要其他内容）：
{
  "genre": "romance|adventure|mystery",
  "world_setting": "一句话世界观描述",
  "protagonist": {"name": "...", "role": "...", "trait": "..."},
  "tone": "叙事风格描述",
  "key_characters": [{"name": "...", "role": "...", "relation": "..."}],
  "central_conflict": "核心冲突描述"
}
```

#### D. 共创阶段：生成大纲树

```
根据以下故事设定，生成大纲树（节点链）。

故事设定：
{story_config_json}

状态变量：
{state_vars_description}

请输出如下格式的 JSON（仅 JSON）：
[
  {
    "node_id": "ch1_xxx",
    "title": "章节标题",
    "goal": "本章叙事目标（给AI看，不透露给玩家）",
    "branches": [
      {"condition": "变量>=值（或空字符串表示无条件）", "next_node": "ch2_xxx"}
    ]
  },
  ...
]

规则：
1. 总节点数控制在 5-8 个
2. 每个节点最多 2 个分支
3. 最后一个节点 branches 为空（即结局节点）
4. 分支条件必须引用已定义的状态变量
5. 节点标题隐晦，不要剧透关键情节
```

## Phase 2: 动态系统与增强交互

> Phase 1 稳定后逐步引入。

### 2.1 动态状态定义

- 移除硬编码模板，共创阶段由 LLM 根据题材**自动生成状态变量**
- 变量定义格式与 Phase 1 相同，只是生成方从"预设文件"变为"LLM 输出"
- 需增加额外的**校验步骤**：检查 LLM 生成的变量定义是否合理（变量名不重复、类型合法等）

### 2.2 Web 界面

- 使用 **FastAPI + SSE** 替换 CLI
- 前端：轻量 HTML/JS，支持：
  - 打字机效果流式渲染
  - 选项按钮点击
  - 侧边栏：状态面板 + 日志 + 设置
  - 移动端适配
- Phase 1 的 `game_loop.py` 核心逻辑保留，只替换 `ui.py`

### 2.3 向量记忆

- 引入 `sentence-transformers` 或 API embedding
- 将角色、地点、道具、关键事件 embed 后存 SQLite + `sqlite-vec`
- 每轮检索最相关条目拼入 Prompt
- 替代 Phase 1 的纯摘要模式

### 2.4 多模型支持

- 抽象 LLM 接口层（见 §4.4 原需求）
- 支持为不同任务配置不同模型：
  - 叙事：主力模型（高创意）
  - 审查/摘要：便宜模型（低成本）

### 2.5 自定义输入

- 实装 §6.2 的自定义输入管线
- 合理性检查模型判断玩家输入是否越界
- 合理的输入转化为"玩家行动描述"注入 Prompt

### 2.6 一致性检查

- 实现可选的审查者模型（§5.2）
- 检查新剧情与大纲的一致性
- 状态变更合理性验证

### 2.7 打字机效果（§6.3）

- Phase 1 使用整段接收后逐句输出的模式
- Phase 2 实现真正的流式打字机效果：边接收边渲染
- 配合 SSE 实现低延迟的逐字展示

### 2.8 存档机制细化（§8）

- 确定存档策略：自动/手动、单槽/多槽
- 实现存档文件版本兼容（schema 迁移机制）
- 优化存档 IO 性能

### 2.9 回档记忆隔离（§3）

- 从 checkpoint 回档后，之前路线的对话历史不混入当前摘要
- 保留所有路线的冒险日志用于结局回顾

### 2.10 摘要生成体验优化（§4.1）

- 摘要触发时显示进度提示
- 考虑异步后台生成摘要，不阻塞叙事请求
- 利用便宜模型做摘要以降低成本

### 2.11 API 调用容错增强（§9）

- 自动重试（可配置次数和间隔）
- 请求失败时的优雅降级策略
- 网络波动时的断点续传

## Phase 3: 完整体验

> 长期愿景，在 Phase 2 成熟后按需启动。

### 3.1 图像生成集成（§7）

- 异步调用 Stable Diffusion / DALL·E API
- 关键场景自动/手动触发生成插画
- 图片嵌入对话流
- 每日/每回合生成上限

### 3.2 云同步

- 存档加密上传至云端
- 跨设备同步游戏进度
- 需明确隐私政策

### 3.3 语音合成（TTS）

- 可选朗读叙事文本
- 不同角色不同声线（如有预算）

### 3.4 剧本导出

- 将冒险历史格式化为小说/剧本格式
- 支持 Markdown / PDF 导出

### 3.5 多人模式

- 不同玩家扮演不同角色
- AI 居中叙述，协调互动
