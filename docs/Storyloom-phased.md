# Storyloom: AI 交互式文字小说游戏引擎（分阶段实施）

> **原完整需求文档**：`docs/Storyloom.md`（保留作为远期参考）
> **当前文档**：将完整愿景拆分为三个可独立交付的阶段，Phase 1 为重点。

## 项目总览

**一句话定位**：以 LLM 为叙事大脑、程序为流程管理器的 AI 交互式文字冒险游戏引擎。

```
Phase 1 ──── CLI 纯文本 MVP ──── 单模型、硬编码状态、规范自然语言输出 ──── 🎯 当前
Phase 2 ──── Web UI + 动态系统 ── 动态状态、向量记忆、多模型
Phase 3 ──── 完整体验 ────────── 图像生成、云同步、TTS、导出
```

## Phase 1: 纯文本单模型 MVP

> **核心原则**：纯文本、单一 API key、不做 token 预算控制、最快跑通完整游戏循环。

### 1.1 目标与范围

#### Phase 1 是什么

- 一个 **终端 CLI** 文字冒险游戏，用户通过键盘选择选项推进故事
- 使用 **单一 OpenAI 兼容 API**（一个 key 覆盖所有 LLM 调用）
- **三套硬编码状态模板**（恋爱/冒险/悬疑），用户选题材即载入
- **大纲树 + 剧情段**：大纲为节点树，每轮 LLM 生成一个"剧情段"，推动故事前进。大纲节点可包含分支，分支内容在同一剧情段内预生成（短片段）
- **极简上下文**：每轮 Prompt 仅携带大纲树 + 剧情进度 + 状态快照 + **上一轮尾部衔接片段**（下文详述），不做历史累积、不做摘要
- **单槽自动存档**：到达关键节点（checkpoint）时自动覆盖保存

#### Phase 1 不是什么

- ❌ 没有图像生成（Stable Diffusion / DALL·E）
- ❌ 没有 Web 界面（纯终端）
- ❌ 没有向量数据库 / embedding 检索
- ❌ 没有多模型调度（叙事、追问、大纲生成用同一模型）
- ❌ 没有自定义文本输入（仅固定选项，自定义输入留到 Phase 2）
- ❌ 不做主动 token 计数与预算控制
- ❌ 不做对话历史摘要（上下文模型不依赖历史累积）

### 1.2 技术选型

| 层面 | 选择 | 说明 |
|------|------|------|
| 语言 | Python 3.10+ | 标准库丰富，生态成熟 |
| LLM SDK | `openai` (Python) | 兼容 OpenAI / 本地 llama.cpp / 多数代理 |
| 终端 UI | `rich` | Markdown 渲染、Panel 布局、主题色、流式逐字展示 |
| 数据存储 | JSON 文件 | 存档即单个 `.json`，可读可调试 |
| 文本解析 | `re`（正则） | 规范自然语言格式，按 `--- 区块名 ---` 分隔符提取 |
| 配置管理 | `.env` 文件 + `python-dotenv` | API key 不进仓库 |

**依赖清单** (`requirements.txt`)：
```
openai>=1.0.0
rich>=13.0.0
python-dotenv>=1.0.0
```

**配置项** (`.env`)：
```
STORYLOOM_API_KEY=sk-xxx
STORYLOOM_API_BASE=https://api.openai.com/v1
STORYLOOM_MODEL=gpt-4o
```

启动时检查 `STORYLOOM_API_KEY` 是否存在，不存在则打印友好提示并退出。Phase 1 不支持游戏内切换模型。

### 1.3 核心架构

单进程事件驱动，模块划分如下：

```
用户输入 → UI (rich) → GameLoop → PromptBuilder → APIClient (stream) → Parser → GameState → 循环
```

| 模块 | 职责 |
|------|------|
| `main.py` | 入口：主菜单、新建/继续 |
| `config.py` | 读取 `.env`，管理模型参数 |
| `api_client.py` | 封装 OpenAI SDK，支持流式输出 + 超时截断 |
| `game_state.py` | 游戏状态数据模型 + 单槽存档读写 + checkpoint 快照 |
| `prompt_builder.py` | 根据阶段组装 System/User Prompt（共创追问 / 设定生成 / 大纲生成 / 叙事循环 / 结局） |
| `parser.py` | 解析 LLM 返回的规范自然语言格式（按 `--- 区块名 ---` 分隔提取） |
| `game_loop.py` | 共创阶段 + 叙事循环 + 结局处理 |
| `ui.py` | `rich` 终端界面：流式逐字展示、选项面板（含置灰不可选项）、状态侧边栏 |

### 1.4 核心概念：剧情段与上下文模型

#### 剧情段（Narration Segment）

每一轮 LLM 生成的内容称为一个"剧情段"。剧情段不是完整故事，而是一段在合理位置自然停顿的叙事文本。

**剧情段边界控制**（三层机制）：

| 机制 | 说明 |
|------|------|
| LLM 自主截断 | LLM 在 Prompt 中被告知目标长度（约 1000 字），在合适的叙事停顿点自然收尾 |
| 程序超时截断 | 若 LLM 响应时间超过阈值，程序截取已生成的内容作为当前剧情段 |
| 程序长度截断 | 若 LLM 未在合理位置收尾，程序在约 75% 位置插入截断点 |

#### 上下文模型——仅携带"上文衔接片段"

**Phase 1 不做历史累积、不做滑动窗口、不做摘要。** 每轮 Prompt 的组装遵循原文 §二.2 的原则：

> "将当前大纲树 + 剧情进度 + 状态系统状态 + 关键说明 + **部分上文情节**组装成 Prompt"

```
┌────────────────────────────────────────────┐
│  System Prompt（固定部分）                   │
│  - 游戏规则与输出格式                        │
│  - 故事设定                                  │
│  - 大纲树（完整）                            │
│  - 当前节点目标                              │
│  - 当前状态快照                              │
│  - 关键说明（如有）                          │
├────────────────────────────────────────────┤
│  User Message（上文衔接）                    │
│  - 上一轮剧情段的尾部片段                     │
│    （即"下一轮标记"之后的内容）               │
└────────────────────────────────────────────┘
```

**"部分上文情节"的定义**：上一轮剧情段中，从 LLM 放置的"下一轮标记"（或程序截断点）到剧情段末尾之间的内容。这段内容的作用是**无缝衔接**——让 LLM 知道上一轮刚好讲到哪里、当前氛围是什么，从而自然延续。

**为什么不需要历史累积**：大纲树承载"故事该往哪走"，状态系统承载"过去发生了什么的结果"，上文片段承载"此刻正在发生什么"。三者合在一起，LLM 不需要看到 10 轮前的原文就能合理续写。

**剧情段内的分支**：当大纲节点包含分支时，LLM 在同一剧情段内生成多个短分支片段。这些分支片段都很短（只是"即时选择的后果快照"），剧情段边界设在分支点。下一轮的 Prompt 从此处继续。详见 §1.7。

### 1.5 游戏流程

```
┌─────────────────────────────────────────────────────┐
│                    主菜单                            │
│            [1] 新游戏  [2] 继续                       │
└─────────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────┐
│  阶段一：共创（设定 & 大纲）                          │
│                                                      │
│  1. 用户输入一句话想法（"想玩一个赛博朋克爱情故事"）    │
│  2. LLM 逐轮追问，澄清题材、世界观、主角               │
│     - 自由对话式，LLM 认为信息足够时向用户确认         │
│     - 用户随时可以说"开始"触发下一步                   │
│     - 建议不超过 20 个问题（不做强制判断）              │
│  3. 用户选择题材模板 → 程序加载对应状态变量             │
│  4. LLM 静默生成故事设定（用户等待，不透露设定内容）    │
│  5. LLM 静默生成大纲树（用户等待）                     │
│  6. 直接进入游戏（大纲不向用户展示，不要求确认）        │
└─────────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────┐
│  阶段二：叙事循环                                     │
│                                                      │
│  循环 {                                              │
│    1. PromptBuilder 组装上下文                        │
│       （大纲 + 进度 + 状态 + 上文衔接片段）            │
│    2. APIClient 流式请求 LLM                         │
│    3. UI 逐字展示正文（流式渲染）                      │
│    4. Parser 解析结构化输出                            │
│    5. UI 展示选项面板                                  │
│       - 满足条件的选项：正常可选                       │
│       - 不满足条件的选项：置灰显示，标注所需条件        │
│    6. 玩家选择选项 (1-5)                              │
│    7. 应用状态变更到 GameState（严格校验）              │
│    8. 若含 checkpoint → 自动存档 + 推进大纲节点        │
│    9. 保存上文衔接片段，准备下一轮                     │
│  }                                                    │
│  结局检测（下一轮 LLM 自然返回结束标记）→ 跳出循环     │
└─────────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────┐
│  阶段三：结局 & 回顾                                  │
│                                                      │
│  1. AI 生成结局段落                                   │
│  2. 生成冒险日志（章节摘要 + 关键决策回顾）            │
│  3. 展示回档选项（所有 checkpoint 节点）               │
│  4. 返回主菜单                                        │
└─────────────────────────────────────────────────────┘
```

#### 结局检测机制

不依赖程序做特殊判断。LLM 在下一轮 Prompt 中看到"大纲已走完 + 上文已达结尾"，自然返回结束标记（`--- 结束 ---`）。程序将其作为正常的一轮处理——只是这一轮恰好是结局段落。三种触发方式均可：

1. LLM 主动返回 `--- 结束 ---`（大纲走完后自然触发）
2. 大纲树最后一个节点完成（`branches` 为空且当前节点已生成 checkpoint）
3. 玩家按 `Q` 键主动结束

### 1.6 状态系统（硬编码模板）

Phase 1 不要求 LLM 动态定义状态变量，而是预设三套模板。用户在共创阶段选择题材后自动载入。

#### 模板 A：恋爱

```json
{
  "genre": "romance",
  "vars": {
    "好感度": {"value": 0, "type": "int", "min": -100, "max": 100},
    "信任度": {"value": 0, "type": "int", "min": 0, "max": 100},
    "当前情绪": {"value": "平静", "type": "enum", "options": ["平静", "喜悦", "悲伤", "愤怒", "紧张"]},
    "关系阶段": {"value": "初识", "type": "enum", "options": ["初识", "朋友", "暧昧", "恋人", "疏远"]}
  }
}
```

#### 模板 B：冒险

```json
{
  "genre": "adventure",
  "vars": {
    "体力": {"value": 100, "type": "int", "min": 0, "max": 100},
    "金币": {"value": 500, "type": "int", "min": 0, "max": 999999},
    "声望": {"value": 0, "type": "int", "min": -100, "max": 100},
    "背包": {"value": [], "type": "list"},
    "队伍": {"value": [], "type": "list"}
  }
}
```

#### 模板 C：悬疑

```json
{
  "genre": "mystery",
  "vars": {
    "理智值": {"value": 100, "type": "int", "min": 0, "max": 100},
    "线索": {"value": [], "type": "list"},
    "时间进度": {"value": "第1天 上午", "type": "string"},
    "嫌疑度": {"value": 0, "type": "int", "min": 0, "max": 100}
  }
}
```

#### 状态更新规则

- LLM 在 `--- 状态 ---` 区块中声明变更，每行一条，格式：`变量名 操作 值`
- 操作类型：
  - `+数值` / `-数值`：int 类型加减
  - `=值`：直接设置（enum、string 类型）
  - `+物品名` / `-物品名`：list 类型追加/移除
- 程序**严格校验**：
  - 变量名必须存在于当前模板
  - `int` 类型不能超出 `min`/`max`
  - `enum` 类型的值必须在 `options` 列表中
  - `list` 类型支持追加和移除
- 校验失败 → 丢弃该条变更，记日志，继续执行（不中断游戏）

#### 状态注入格式

每轮 Prompt 中注入当前状态摘要：

```
当前状态：好感度 30 | 信任度 45 | 情绪 平静 | 关系阶段 朋友
```

### 1.7 结构化输出规范（规范自然语言）

**核心设计决策**：Phase 1 不使用 JSON 或 XML。所有 LLM 输出采用"规范自然语言 + 区块分隔符"格式。LLM 生成自然语言比生成严格 JSON/XML 更稳定，程序用正则按分隔符提取各区块即可。

#### A. 故事设定（共创阶段 → 程序）

```
=== 故事设定 ===
题材：恋爱
世界观：2077年新东京，赛博朋克都市
主角姓名：林晓
主角身份：黑客
主角特质：内向但正义感强
叙事风格：冷峻与温情交织
核心冲突：AI与人类的身份认同危机
主要角色：
  - AIKO | AI助手 | 主角的搭档
  - 田中社长 | 企业总裁 | 幕后黑手
```

**解析规则**：
- 顶格字段（`题材：`、`世界观：` 等）→ 正则 `^字段名：(.+)$`
- 缩进列表项（`  - ...`）→ 按 `|` 分割为 name/role/relation
- 字段缺失 → 重试（附带缺失字段提示），最多 2 次

#### B. 大纲树（共创阶段 → 程序）

```
=== 大纲树 ===
节点1：神秘委托 | ch1_intro
  目标：主角接到一个无法拒绝的黑客任务
  分支：无

节点2：真相浮现 | ch2_discovery
  目标：发现委托人其实是AI
  分支：信任度>=50 → ch3_ally
  分支：信任度<50 → ch3_betrayal

节点3：联手 | ch3_ally
  目标：与AIKO联手对抗企业
  分支：无

节点4：背叛 | ch3_betrayal
  目标：独自面对真相
  分支：无

节点5：结局 | ch_final
  目标：所有线索汇集，命运交汇
  分支：无
```

**解析规则**：
- `节点N：标题 | node_id` → 提取节点信息
- `目标：...` → 节点目标描述
- `分支：条件 → 目标节点` 或 `分支：无`
- 总节点数建议 5-8 个，每个节点最多 2 个分支

**静态校验**：大纲树生成后，程序检查所有分支条件引用的变量是否存在于当前状态模板。校验失败 → 重试（附带变量名列表），最多 2 次。

#### C. 叙事输出（游戏循环 LLM → 程序）

```
--- 正文 ---
夜幕降临，新东京的霓虹灯在雨雾中晕开一片迷离的光。

你推开地下酒吧的门，潮湿的空气混杂着合成酒精和烧焦电路的气味扑面而来。吧台深处的角落里，一个戴着全息面具的人朝你招了招手。

--- 选项 ---
1. 径直走过去，开门见山谈正事
2. 先在吧台点一杯酒，观察周围 @需要:理智值>=50
3. 假装没看见，转身离开

--- 状态 ---
理智值 -5
线索 +神秘人的全息面具

--- 节点 ---
主要节点 ch2_discovery
```

**区块说明**：

| 区块 | 必需 | 说明 |
|------|------|------|
| `--- 正文 ---` | ✅ 必选 | 故事叙述正文。LLM 目标约 1000 字，程序流式逐字展示。LLM 可在自然段间用空行分隔以控制展示节奏 |
| `--- 选项 ---` | 可选 | 2-5 个选项，每行 `N. 描述`；可选条件标记 `@需要:变量>=值`。不满足条件的选项**置灰显示**（不可选），不是隐藏 |
| `--- 状态 ---` | 可选 | 每行一条变更：`变量名 操作 值`。操作：`+N`/`-N`（加减）、`=值`（设置）、`+物品`/`-物品`（list 操作） |
| `--- 节点 ---` | 可选 | 到达大纲关键节点时标记。格式：`主要节点 node_id` 或 `结束节点`（结局触发） |
| `--- 续 ---` | 可选 | LLM 放置的"下一轮标记"。此标记之后的内容作为下一轮的上文衔接片段。若 LLM 未放置，程序在总长度约 75% 处截断 |

**分支剧情段**：当大纲节点包含分支时，正文中在各分支选项后附带短分支片段：
```
--- 正文 ---
（分支前正文……）

你面临抉择——信任这个神秘的委托人，还是独自调查？

（若选"信任委托人"）你握住了他伸出的手，冰冷的金属触感让你心头一颤……
（若选"独自调查"）你转身消失在雨夜中，身后传来一声叹息……

--- 选项 ---
1. 信任委托人
2. 独自调查

--- 续 ---
（无缝衔接到下一轮的过渡内容）
```

分支片段必须简短（每个分支 50-100 字），只是"即时后果快照"。剧情段边界（`--- 续 ---`）设在分支之后。下一轮从此处继续。

#### D. 结束标记

```
--- 结束 ---
（可选的结局叙述段落）
```

程序检测到 `--- 结束 ---` 区块即进入结局流程。

### 1.8 解析流程与错误处理

#### 解析流程

```
LLM 原始响应
  → 预处理：strip markdown 代码块围栏（```...```）
  → 正则按 `^--- .+ ---$` 分割为区块
  → 逐区块解析：
      "正文" → 提取纯文本 → 送 UI 流式展示
      "选项" → 逐行解析 "N. 描述 @需要:条件" → 构建选项列表
      "状态" → 逐行解析 "变量 操作 值" → 校验 → 更新 GameState
      "节点" → 提取 node_id → 推进大纲 → 触发存档
      "续"   → 保存为下一轮的上文衔接片段
      "结束" → 触发结局流程
```

#### 错误处理策略

| 场景 | 处理 |
|------|------|
| `--- 正文 ---` 缺失 | 重试（附带格式纠正提示），最多 2 次。两次均失败 → 若有上一次有效响应则回退；若无（首轮）→ 告知用户，询问重试或返回主菜单 |
| `--- 选项 ---` 缺失 | 静默跳过。仅展示正文，不显示选项面板。下一轮自动推进（程序发送空选择 + 上文片段） |
| `--- 选项 ---` 中个别行解析失败 | **严格丢弃**该条目，只展示解析成功的选项。若全部失败 → 等同选项缺失 |
| `--- 状态 ---` 中个别行校验失败 | 丢弃该条变更，记日志，继续执行。不影响其他有效变更 |
| `--- 节点 ---` 中的 node_id 在大纲中不存在 | 记日志，忽略该节点标记。不推进大纲，不触发存档 |
| `--- 续 ---` 缺失且内容超 1000 字 | 程序在 75% 位置截断，截断点后的内容作为上文衔接片段 |
| LLM 响应超时 | 截取已生成内容，在最后一个完整段落后插入截断点 |

#### 共创阶段的错误处理

| 场景 | 处理 |
|------|------|
| 故事设定解析失败（字段缺失） | 重试（附带缺失字段列表），最多 2 次。两次均失败 → 告知用户并返回主菜单 |
| 大纲树解析失败 | 重试（附带格式纠正提示），最多 2 次。两次均失败 → 告知用户并返回主菜单 |
| 大纲树静态校验失败（分支条件引用不存在的变量） | 重试（附带可用变量名列表），最多 2 次 |

### 1.11 选项条件系统

#### 格式

LLM 在选项中附带条件标记：
```
2. 先在吧台点一杯酒，观察周围 @需要:理智值>=50
```

#### 程序处理

1. 解析选项描述和条件（`@需要:变量 运算符 值`）
2. 支持的运算符：`>=`, `<=`, `>`, `<`, `==`
3. 每轮展示前，用当前 GameState 评估每个选项的条件
4. 满足条件 → 正常可选
5. 不满足条件 → **置灰显示**（`rich` 中可用 dim 样式），选项文字后标注 `（需要：理智值>=50，当前：30）`
6. 若全部选项都不满足条件 → 全部正常展示（兜底策略，防止玩家卡死）
7. 若条件中引用的变量名不存在 → 视为无条件（不阻断选项），记日志

#### 示例展示效果

```
  [1] 径直走过去，开门见山谈正事
  [2] 先在吧台点一杯酒，观察周围 （需要：理智值>=50，当前：30）
  [3] 假装没看见，转身离开
```

选项 2 显示但无法选择（按键无响应或显示提示）。

## Phase 2: 动态系统与增强交互

> Phase 1 稳定后逐步引入。

### 2.1 动态状态定义

- 移除硬编码模板，共创阶段由 LLM 根据题材**自动生成状态变量**
- 变量定义格式与 Phase 1 相同，只是生成方从"预设文件"变为"LLM 输出"
- 需增加额外的**校验步骤**：检查 LLM 生成的变量定义是否合理（变量名不重复、类型合法等）

### 2.2 Web 界面

- 使用 **FastAPI + SSE** 替换 CLI
- 前端：轻量 HTML/JS，支持：
  - 打字机效果流式渲染（真正的逐字 SSE）
  - 选项按钮点击
  - 侧边栏：状态面板 + 日志 + 设置
  - 移动端适配
- Phase 1 的 `game_loop.py` 核心逻辑保留，只替换 `ui.py`

### 2.3 向量记忆

- 引入 `sentence-transformers` 或 API embedding
- 将角色、地点、道具、关键事件 embed 后存 SQLite + `sqlite-vec`
- 每轮检索最相关条目拼入 Prompt
- 作为当前"部分上文"上下文模型的增强——在 System Prompt 中增加"相关背景"区块

### 2.4 多模型支持

- 抽象 LLM 接口层（见 §4.4 原需求）
- 支持为不同任务配置不同模型：
  - 叙事：主力模型（高创意）
  - 审查/追问：便宜模型（低成本）

### 2.5 自定义输入

- 实装 §6.2 的自定义输入管线
- 合理性检查模型判断玩家输入是否越界
- 合理的输入转化为"玩家行动描述"注入 Prompt

### 2.6 一致性检查

- 实现可选的审查者模型（§5.2）
- 检查新剧情与大纲的一致性
- 状态变更合理性验证

### 2.7 存档机制增强

- 多槽位支持（3-5 个槽位，可命名）
- 手动存档（玩家按键触发）
- 存档文件版本兼容（schema 迁移机制）

### 2.8 回档记忆隔离

- 从 checkpoint 回档后，之前路线的对话历史不混入当前上下文
- 保留所有路线的冒险日志用于结局回顾

### 2.9 上下文增强

- 当前 Phase 1 的极简上下文模型（仅上文衔接片段）在长线游戏中可能丢失早期细节
- 引入向量记忆（2.3）作为补充
- 引入可选的历史摘要（仅当上下文明显不足时触发）
- 异步后台生成摘要，不阻塞叙事请求

### 2.10 API 调用容错增强

- 自动重试（可配置次数和间隔）
- 请求失败时的优雅降级策略
- 网络波动时的断点续传
