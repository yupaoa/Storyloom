# Storyloom 设计概要

> 程序执行逻辑详见 [`Phase1-exec-flow.md`](./Phase1-exec-flow.md)。本文档表达设计思路与阶段规划，不涉及实现细节。

---

## 项目定位

以 LLM 为叙事大脑、程序为流程管理器的交互式文字冒险游戏引擎。分三阶段交付：

```
Phase 1 ──── CLI 纯文本 MVP ──── 验证核心游戏循环
Phase 2 ──── Web + 动态系统 ──── 从工具迈向产品
Phase 3 ──── 完整体验 ────────── 图像 / 语音 / 云同步
```

---

## Phase 1 设计

### 整体流程

```
主菜单 → 共创 → 叙事循环(N轮) → 结局 → 主菜单
```

### 共创阶段

用户输入初始想法，LLM 追问完善五大维度：**世界观、主角设定、故事基调、冲突方向、故事长度**。确认后静默生成两份数据：

- **story_config** — 故事设定（题材、角色、世界观等元数据）
- **outline** — 大纲有向图（节点含叙事目标 + 可选的条件分支）

题材映射到三套状态模板之一（恋爱/冒险/悬疑），载入初始 state_vars。大纲和设定的具体内容不向玩家展示，保持新鲜度。

### 叙事循环

每轮 LLM 生成一个"剧情段"，是游戏推进的基本单位。核心机制：

**bridge 预加载** — 实现无缝体验的关键。LLM 在段中插入 `--- bridge ---`：程序执行到此处立即提交下一轮 Prompt，同时继续展示 bridge 之后的缓冲叙事。玩家读到段尾时，下一段已生成完毕，体感无等待。

**极简上下文** — 每轮 Prompt 只携带五类信息，不做对话历史累积：

| 信息 | 作用 |
|------|------|
| 大纲树 + 节点进度 | "故事该往哪走" |
| 状态快照 | "过去发生了什么的结果" |
| checkpoint 摘要列表 | "已发生的关键情节" |
| bridge_text | "此刻正在发生什么" |
| 被拒变更反馈 | "上次哪些状态变更不合法" |

**区块分隔符** — LLM 输出采用 `--- block_name ---` 结构化标记，程序按正则分割后逐区块解析执行：

| 区块 | 职责 |
|------|------|
| `narrative` | 叙事正文，支持命名分支（`narrative:branch_a`）实现段内变体 |
| `options` | 玩家选项，支持 `@if` 条件置灰和 `-> branch` 路由 |
| `state` | 状态变更建议 + `@branch` 段内路由 |
| `checkpoint` | 大纲节点推进 + `if → route` 大纲分支 |
| `bridge` | 预加载触发点，bridge 之后仅允许 narrative |
| `adventure_log` | 结局冒险回顾（独立 Prompt 生成） |

### 两层分支模型

**段内分支** — 不影响大纲走向。LLM 输出多个命名 narrative（如 `narrative:took_chip`、`narrative:left`），玩家选择通过 `@branch` 切换 current_branch，程序只展示/提取匹配分支的内容。同段各分支在 bridge 处汇合，下一轮统一从 `narrative:main` 开始。

**大纲分支** — 改变故事路线。checkpoint 中声明 `if 条件 -> route 目标节点`，程序用本地状态评估条件，推进到不同的大纲节点。各分支后续独立发展，不汇合。

### 状态系统

三套硬编码模板定义状态变量的类型、初始值和约束。LLM 通过 `--- state ---` 建议变更（加减数值、设置枚举、增删列表元素）；程序逐条校验——变量存在、类型匹配、范围合规——通过后方可应用。被拒变更在下轮反馈给 LLM。

核心理念：**本地 GameState 为唯一真相源**。LLM 只能建议，程序拥有最终控制权。

### 存档

每局游戏一个独立 JSON 文件（`saves/{label}.json`），到达 checkpoint 时原子写入（先写 `.tmp` 再 `os.replace`）。存档自包含——创建后不受模板文件后续变更影响。加载时校验版本号、关键字段完整性、模板标识存在性，任一失败则判定存档损坏。

### 结局

两条路径汇聚：checkpoint `end`（自然结局）或玩家按 Q（主动结束）。结局时独立生成冒险日志——章节摘要、关键抉择回顾、结局评语——不走正常叙事循环管线。

## Phase 2 设计

从 CLI 工具进化到 Web 产品，核心变化：

| 能力 | 变化 |
|------|------|
| **动态状态** | LLM 根据题材自动生成状态变量，不再依赖硬编码模板 |
| **Web 界面** | FastAPI + SSE 流式渲染，选项按钮、状态面板、移动端适配 |
| **向量记忆** | 角色/地点/事件 embed 存储，每轮检索注入 Prompt，弥补无历史累积的局限 |
| **多模型** | 叙事用主力模型，审查/追问用便宜模型，降本增效 |
| **自定义输入** | 玩家可自由输入行动，合理性检查模型把关 |
| **一致性审查** | 独立模型检查剧情与大纲、状态变更的合理性 |

Phase 1 核心逻辑（`game_loop.py`）保留，仅替换 UI 层。

---

## Phase 3 设计

完整体验层，在 Phase 2 成熟后按需启动：

- **图像生成** — 关键场景异步生成插画，嵌入对话流
- **云同步** — 存档加密上传，跨设备同步进度
- **TTS** — 可选角色语音朗读叙事文本
- **剧本导出** — 冒险历史格式化为 Markdown/PDF 剧本
- **多人模式** — 不同玩家扮演不同角色，AI 居中叙述协调

---

## 设计原则

| 原则 | 说明 |
|------|------|
| **LLM 叙事，程序控流** | LLM 负责创意内容；程序负责数据完整性、流程控制、状态校验 |
| **本地唯一真相源** | 一切游戏数据以本地 GameState 为准，LLM 只能建议变更 |
| **预加载无缝衔接** | bridge 机制确保下一段在用户阅读当前段时已后台生成 |
| **失败透明** | API 错误、解析失败告知用户并等待决策，不静默回退 |
| **错误隔离** | 单条 state 失败不影响同段其他变更；单选项解析失败不影响其余选项 |
