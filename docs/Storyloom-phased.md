## 项目总览

**一句话定位**：以 LLM 为叙事大脑、程序为流程管理器的 AI 交互式文字冒险游戏引擎。

```
Phase 1 ──── CLI 纯文本 MVP ──── 单模型、硬编码状态、规范自然语言输出 ──── 🎯 当前
Phase 2 ──── Web UI + 动态系统 ── 动态状态、向量记忆、多模型
Phase 3 ──── 完整体验 ────────── 图像生成、云同步、TTS、导出
```

## Phase 1: 纯文本单模型 MVP

> **核心原则**：纯文本、单一 API key、不做 token 预算控制、最快跑通完整游戏循环。

### 1.1 目标与范围

#### Phase 1 是什么

- 一个 **终端 CLI** 文字冒险游戏，用户通过键盘选择选项推进故事
- 使用 **单一 OpenAI 兼容 API**（一个 key 覆盖所有 LLM 调用）
- **三套硬编码状态模板**（恋爱/冒险/悬疑），用户选题材即载入
- **大纲树 + 剧情段**：大纲为节点树，每轮 LLM 生成一个"剧情段"，推动故事前进。大纲节点可包含分支，分支内容在同一剧情段内预生成（短片段）
- **极简上下文**：每轮 Prompt 携带大纲树 + 剧情进度 + 状态快照 + Checkpoint 摘要列表 + 上一轮尾部衔接片段，不做对话历史累积
- **单槽自动存档**：到达关键节点（checkpoint）时自动覆盖保存

#### Phase 1 不是什么

- ❌ 没有图像生成（Stable Diffusion / DALL·E）
- ❌ 没有 Web 界面（纯终端）
- ❌ 没有向量数据库 / embedding 检索
- ❌ 没有多模型调度（叙事、追问、大纲生成用同一模型）
- ❌ 没有自定义文本输入（仅固定选项，自定义输入留到 Phase 2）
- ❌ 不做主动 token 计数与预算控制
- ❌ 不做对话轮次历史累积（上下文模型不依赖逐轮历史，仅用 checkpoint 摘要替代）

### 1.2 技术选型

| 层面 | 选择 | 说明 |
|------|------|------|
| 语言 | Python 3.10+ | 标准库丰富，生态成熟 |
| LLM SDK | `openai` (Python) | 兼容 OpenAI / 本地 llama.cpp / 多数代理 |
| 终端 UI | `rich` | Markdown 渲染、Panel 布局、主题色、程序控制逐段展示 |

| 配置管理 | `.env` 文件 + `python-dotenv` | API key 不进仓库 |

```
openai>=1.0.0
rich>=13.0.0
python-dotenv>=1.0.0
```

**配置项** (`.env`)：
```
STORYLOOM_API_KEY=sk-xxx
```

启动时检查 `STORYLOOM_API_KEY` 是否存在，不存在则打印友好提示并退出。Phase 1 不支持游戏内切换模型。

### 1.3 核心架构

单进程事件驱动，模块划分如下：

```
用户输入 → UI (rich) → GameLoop → PromptBuilder → APIClient (stream) → Parser → GameState → 循环
```

| 模块 | 职责 |
|------|------|
| `main.py` | 入口：主菜单、新建/继续 |
| `config.py` | 读取 `.env`，管理模型参数 |
| `api_client.py` | 封装 OpenAI SDK，支持流式输出 + 超时截断 |
| `game_state.py` | 游戏状态数据模型 + 单槽存档读写 + checkpoint 快照 |
| `prompt_builder.py` | 根据阶段组装 System/User Prompt（共创追问 / 设定生成 / 大纲生成 / 叙事循环 / 结局） |
| `parser.py` | 解析 LLM 返回的规范自然语言格式（按 `--- 区块名 ---` 分隔提取） |
| `game_loop.py` | 共创阶段 + 叙事循环 + 结局处理 |
| `ui.py` | `rich` 终端界面：程序控制逐段展示正文、选项面板（含置灰不可选项）、状态侧边栏 |

### 1.4 核心概念：剧情段、上下文模型、分支与节点推进

#### 1.4.1 剧情段（Narration Segment）

每一轮 LLM 生成的内容称为一个"剧情段"。剧情段不是完整故事，而是一段在合理位置自然停顿的叙事文本，包含正文叙述及可选的选项、状态变更、节点标记等结构化区块。

**剧情段边界控制**（三层机制，均使用可配置常量，不硬编码）：

| 机制 | 触发条件 | 行为 | 参考值 |
|------|---------|------|--------|
| LLM 自主截断 | LLM 在合适叙事停顿点自然收尾 | 理想情况，无需程序干预 | 目标 ~1000 字 |
| 程序长度截断 | LLM 生成的正文超过上限 | 在最后一个完整段落（以空行或句号/问号/感叹号结尾）处截断，截断点之后的内容作为 bridge_text | 参考值 2000 字 |
| 流式停顿超时 | 用户已读完当前内容，LLM 流式输出停顿过久 | 截取已生成内容，在最后一个完整段落后截断 | 参考值 5 秒 |

**内容过少处理**：若截取内容低于最低阈值（参考值 100 字），判定为异常，提示用户确认是否重试或返回主菜单。

#### 1.4.2 上下文模型

Phase 1 不做历史累积、不做滑动窗口。每轮 Prompt 的上下文由以下层次组成：

```
┌────────────────────────────────────────────┐
│  System Prompt（固定部分）                   │
│  - 游戏规则与输出格式                        │
│  - 故事设定（story_config）                  │
│  - 大纲树（完整 outline tree）               │
│  - 剧情进度（当前节点 + 节点状态）            │
│  - 当前节点目标（goal）                      │
│  - 当前状态快照（所有 state_vars）            │
│  - Checkpoint 摘要列表（累积，见下方）        │
├────────────────────────────────────────────┤
│  User Message（上文衔接）                    │
│  - bridge_text：上一轮留下的衔接片段          │
│    （LLM 放置的 --- 续 --- 内容，或程序截断点  │
│     之后的内容）                              │
└────────────────────────────────────────────┘
```

**"部分上文情节"（bridge_text）的定义**：
- 若 LLM 在上一轮输出了 `--- 续 ---` 区块，该区块内容即为 bridge_text
- 若 LLM 未放置 `--- 续 ---` 但正文超长被程序截断，截断点之后的内容即为 bridge_text
- 若正文未超长且无 `--- 续 ---`，bridge_text 为空
- bridge_text 的作用是**无缝衔接**——让 LLM 知道上一轮刚好讲到哪里、当前氛围是什么，从而自然延续

**为什么不需要历史累积**：大纲树承载"故事该往哪走"，状态系统承载"过去发生了什么的结果"，上文片段承载"此刻正在发生什么"，checkpoint 摘要承载"已发生的关键情节"。四者合在一起，LLM 不需要看到多轮前的原文即可合理续写。

**Checkpoint 摘要**：为弥补无历史累积的局限，到达关键节点时 LLM 在 `--- 节点 ---` 区块中附带一段摘要（从上一 checkpoint 到当前节点的情节概述）。程序将此摘要存入 save 的 `checkpoint_summaries` 列表，每轮注入 System Prompt，结局时作为冒险日志的生成素材。

#### 1.4.3 分支分类与处理

分支分为两类，处理方式不同：

**局部小分支（非关键节点内的小分支）**
- 用途：提升沉浸感的剧情微分支（如不同对话回应），不改变大纲走向
- LLM 输出方式：在正文之后放置 `--- 分支 ---` 区块，每个分支用 `[N]` 标记对应选项编号
- 每个分支片段 50-100 字，只是"即时后果快照"
- 程序处理：展示选项 → 玩家选择 → **只展示对应编号的分支片段**，隐藏其余分支
- 小分支可合并：`--- 续 ---` 放置在所有分支片段**之后**，作为通用的上文衔接（即小分支不影响主线走向）
- 注意：`--- 分支 ---` 区块内的内容**不展示给玩家**直到其做出选择，避免剧透

**关键节点分支（大纲层面的路线分叉）**
- 用途：大纲中标记了多个后续节点的位置，改变故事走向
- LLM 在到达该节点时，在 `--- 节点 ---` 区块中输出可判定的分支条件（引用状态变量）
- 程序解析条件 → 本地评估当前 GameState → 确定下一节点 → 更新 current_node 和大纲进度
- 关键节点分支**禁止在同一个 `--- 续 ---` 中合并**——各分支走各自的路线，后续剧情段独立发展

#### 1.4.4 节点推进逻辑（核心数据流）

这是游戏引擎最核心的状态机逻辑。每一步的数据以本地 GameState 为准，LLM 只负责叙事和标记，不直接修改程序数据。

**节点推进的完整流程**：

```
1. 程序组装 Prompt（含 current_node、outline、state_vars、bridge_text、checkpoint 摘要列表）
      ↓
2. LLM 生成剧情段，若到达大纲关键节点则放置 --- 节点 --- 区块
      ↓
3. 程序解析到 --- 节点 --- 区块：
   a. 提取 node_id（LLM 标记到达了哪个大纲节点）
   b. 验证 node_id 是否在大纲中存在（不存在 → 记日志，忽略，继续）
   c. 将当前 active 节点标记为 completed
   d. 若 --- 节点 --- 包含"分支条件"，程序解析条件表达式
      - 用当前 GameState 评估每个条件
      - 取第一个满足的条件，其 next_node 即为目标节点
      - 若无条件满足 → 取第一个分支的 next_node（兜底策略）
   e. 若 --- 节点 --- 包含"摘要"，存入 checkpoint_summaries
   f. 将目标节点（若有分支则为匹配的 next_node，若无分支则为大纲中的下一个节点）标记为 active
   g. 保存当前状态快照到 checkpoint_snapshots[node_id]
   h. 触发自动存档
      ↓
4. 下一轮 Prompt 中注入更新后的 current_node 和 outline node status
   → LLM 看到新的当前节点目标，围绕新目标叙事
```

**关于分支条件的说明**：
- 大纲树生成阶段，LLM 给出分支条件的**大致方向**（可引用状态变量，也可仅做文字描述）
- 运行时到达分支节点时，若大纲中的分支条件不够具体，LLM 在 `--- 节点 ---` 区块中输出具体的、程序可判定的条件
- 若大纲中的分支条件已足够具体（引用了状态变量和运算符），程序可直接用大纲数据评估，`--- 节点 ---` 只需 node_id 和摘要
- 分支条件引用的变量名必须是当前模板中存在的变量。若引用不存在的变量 → 记日志，该条件视为无效，跳过，取下一个条件或兜底
- 多个分支条件同时满足时，**取第一个匹配的**（LLM 应按优先级排列条件）
- 分支条件格式：`变量名 运算符 值 → 目标node_id`，如 `信任度>=50 → ch3_ally`

### 1.5 游戏流程（详细实现）

#### 1.5.1 主菜单

```
┌─────────────────────────────────────────────────────┐
│               Storyloom - 文字冒险                    │
│                                                     │
│              [1] 新游戏    [2] 继续                   │
│                                                     │
│  当前状态：无存档                                     │
└─────────────────────────────────────────────────────┘
```

**"新游戏"行为**：
1. 检查 `saves/save.json` 是否存在
2. 若存在 → 提示"已有存档，开始新游戏将覆盖旧存档，是否继续？"
3. 若不存在 → 直接进入共创阶段

**"继续"行为**：
1. 检查 `saves/save.json` 是否存在且 JSON 结构完整
2. 若不存在 → 提示"没有存档，请开始新游戏"
3. 若存在但 JSON 无法解析 → 提示"存档已损坏，请开始新游戏"，删除或忽略损坏文件
4. 若正常 → 加载存档，恢复全部状态，跳入叙事循环

#### 1.5.2 阶段一：共创（设定 & 大纲）——详细步骤

**Step 1: 用户输入初始想法**
- 程序提示："请用一句话描述你想玩的故事（如'赛博朋克背景下的爱情故事'）"
- 用户输入自由文本

**Step 2: 追问对话（同一对话上下文）**
- 程序将用户输入发送给 LLM（追问 Prompt，§1.10-B）
- LLM 以提问方式引导用户完善设定，每个问题可附带 2-3 个参考选项
- 用户回复后，LLM 继续追问下一个方向
- **题材确认**：若用户首轮已暗示题材（如"爱情故事"），LLM 在第一次追问中确认；否则第一个问题就是"你想玩什么类型？恋爱/冒险/悬疑？"
- **追问范围限制**：LLM 不得过度追问细节，不得涉及具体情节内容（保持新鲜度）。聚焦于世界观、主角设定、故事基调、冲突方向
- **终止条件**：
  - LLM 认为信息足够 → 在回复末尾询问"是否开始生成故事？"
  - 用户在任何时候输入"开始"或类似词语 → 停止追问
  - 不做程序性强制作断（不设轮数上限）
- **追问过程中的修正**：用户可在后续对话中自然修正之前的回答（如"改成在太空站吧"），无需回退机制

**Step 3: 生成故事设定（同一对话上下文，静默生成）**
- 用户确认开始后，程序发送设定生成 Prompt（§1.10-C）
- LLM 输出 `=== 故事设定 ===` 格式的结构化设定
- 程序解析题材字段（`romance` / `adventure` / `mystery`），据此加载对应状态模板
- 用户在此期间看到等待提示（如"正在编织故事世界……"），不透露设定具体内容
- 解析失败 → 重试（附带缺失字段列表），最多 2 次。两次均失败 → 告知用户并返回主菜单

**Step 4: 生成大纲树（新对话上下文，静默生成）**
- 若 Step 2+3 的对话上下文过长，使用新对话上下文
- 程序将故事设定 + 可用状态变量列表注入大纲生成 Prompt（§1.10-D）
- LLM 输出 `=== 大纲树 ===` 格式的大纲
- **静态校验**：程序检查大纲树的完整性
  - 所有 `分支条件` 引用的变量是否存在于当前状态模板
  - 所有 `→ 目标node_id` 是否指向大纲中存在的节点
  - 最后一个节点的 `分支` 是否为"无"
  - 校验失败 → 重试（附带错误提示），最多 2 次
- 用户在此期间看到等待提示（如"正在绘制故事脉络……"）

**Step 5: 进入游戏**
- 大纲树**不向用户展示**
- 程序初始化叙事循环所需数据：current_node = 首个节点，bridge_text = ""（首轮为空），checkpoint_summaries = []
- 直接进入叙事循环

#### 1.5.3 阶段二：叙事循环——详细步骤

每一轮的完整执行序列：

```
┌─────────────────────────────────────────────────────┐
│ Round N 开始                                         │
├─────────────────────────────────────────────────────┤
│                                                      │
│  1. PromptBuilder.assemble()                         │
│     输入：story_config, outline, current_node,       │
│           state_vars, bridge_text,                   │
│           checkpoint_summaries                       │
│     输出：system_prompt + user_message               │
│                                                      │
│  2. APIClient.stream_chat(system, user)              │
│     - 流式接收 LLM 原始响应（完整接收后才解析）        │
│     - 若 API 调用失败（网络/rate limit/server error）  │
│       → 告知用户错误信息，询问是否重试                 │
│     - 若流式输出停顿超过配置阈值 → 截取已生成内容       │
│                                                      │
│  3. UI.show_narration(raw_response)                  │
│     - 展示等待动画（如"..."），直到收到首批文本        │
│     - 程序接收完整响应后，解析 --- 正文 --- 区块       │
│     - 按段落/句子逐行或逐段展示正文（程序控制节奏，    │
│       非逐字流式）                                    │
│     - --- xxx --- 标记行不展示给用户                  │
│                                                      │
│  4. Parser.parse(raw_response)                       │
│     → 返回结构化数据：narration, options[],           │
│        state_changes[], node_marker,                  │
│        branch_snippets[], bridge_text, is_ending      │
│     （详见 §1.8 解析流程）                            │
│                                                      │
│  5. UI.show_options(options, game_state)              │
│     - 对每个选项评估 @需要: 条件                       │
│     - 满足 → 正常可选的编号 + 描述                    │
│     - 不满足 → dim 样式置灰，标注"（需要：XX，当前：Y）"│
│     - 全部不满足 → 全部正常展示，移除条件标注          │
│     - 无选项 → 跳过选项面板，自动推进（见步骤 6.5）    │
│                                                      │
│  6. 玩家交互分支                                      │
│     ┌─────────────────────────────────────┐          │
│     │ 6.1 有选项 → 用户按键选择 (1-5)      │          │
│     │ 6.2 有 --- 分支 --- → 只展示对应     │          │
│     │     编号 [N] 的分支片段，隐藏其余     │          │
│     │ 6.3 置灰选项被按下 → 提示不可选      │          │
│     │ 6.4 用户按 Q → 触发主动结束          │          │
│     │ 6.5 无选项 → 自动推进（不等待输入）   │          │
│     └─────────────────────────────────────┘          │
│                                                      │
│  7. GameState.apply_changes(state_changes)            │
│     - 逐条校验（变量存在、类型匹配、范围合规）          │
│     - 校验失败 → 丢弃该条 + 记日志 + 继续下一条        │
│     - 校验成功 → 更新 state_vars                      │
│     - 记录被拒绝的变更及其原因（供下轮 Prompt 告知LLM） │
│                                                      │
│  8. 节点处理（若 --- 节点 --- 存在）                   │
│     a. 提取 node_id                                  │
│     b. 验证 node_id 存在于 outline                    │
│     c. 标记旧 current_node 为 completed               │
│     d. 解析分支条件 → 评估 → 确定目标节点              │
│        （取第一个满足条件的分支，详见 §1.4.4）         │
│     e. 标记目标节点为 active，更新 current_node        │
│     f. 保存 checkpoint 摘要到 checkpoint_summaries    │
│     g. 保存状态快照到 checkpoint_snapshots            │
│     h. 触发自动存档（覆盖 save.json）                  │
│                                                      │
│  9. 准备下一轮                                        │
│     - bridge_text = --- 续 --- 内容（或程序截断内容）  │
│     - round_count += 1                               │
│     - 若 --- 续 --- 缺失且正文未超长 → bridge_text 为空│
│                                                      │
│  10. 结局检测                                         │
│      - 若 --- 结束 --- 存在 → 跳出循环，进入结局阶段   │
│      - 若 Q 键按下 → 跳出循环，进入结局阶段            │
│      - 否则 → 回到步骤 1，开始 Round N+1               │
│                                                      │
└─────────────────────────────────────────────────────┘
```

**无选项时的自动推进**：当 `--- 选项 ---` 缺失时，程序不等待玩家输入，直接将用户未做选择（空选择）的标记注入下一轮的 User Message，LLM 从 bridge_text 自然延续叙事。这用于场景过渡或纯叙事段落。

#### 1.5.4 阶段三：结局 & 回顾

**结局触发**（三种方式，任一种均进入结局流程）：
1. LLM 返回 `--- 结束 ---` 区块（大纲已走完，LLM 从 Prompt 数据自然判断故事结束）
2. 玩家在选项提示时按 `Q` 键主动结束
3. 大纲最后一个节点完成（branches 为空且已生成 checkpoint）后，下一轮 LLM 自然返回结束标记

**结局流程**：
1. 若为 Q 键触发 → 程序发送结局 Prompt（§1.10-E）给 LLM 生成结局段落和冒险日志
2. 若为 `--- 结束 ---` 触发 → 正文即为结局叙述，若 LLM 同时输出 `--- 冒险日志 ---` 则一并展示
3. UI 展示结局正文
4. UI 展示冒险日志（来源：checkpoint_summaries + 最终状态 + 关键节点标题）
5. 展示 checkpoint 节点列表（供未来回档，Phase 1 仅展示不交互）
6. 按任意键返回主菜单

**冒险日志的数据来源**：
- `checkpoint_summaries`：每个 checkpoint 节点的情节摘要（由 LLM 在 `--- 节点 ---` 中生成并累积）
- `checkpoint_history`：已完成的 checkpoint node_id 列表及其标题
- `state_summary`：最终状态快照
- 结局 Prompt 将这些数据注入，LLM 据此生成完整冒险日志

> **注**：Phase 1 不做回档功能。`checkpoint_snapshots` 数据结构保留在存档中为 Phase 2 预留，但当前版本不从 checkpoint 恢复游戏。

### 1.6 状态系统（硬编码模板）

Phase 1 不要求 LLM 动态定义状态变量，而是预设三套模板。模板文件位于 `templates/states.json`，程序在共创阶段解析出题材后自动载入。

#### 模板定义

**模板 A：恋爱**
```json
{
  "genre": "romance",
  "label": "恋爱",
  "vars": {
    "好感度": {"value": 0, "type": "int", "min": -100, "max": 100},
    "信任度": {"value": 0, "type": "int", "min": 0, "max": 100},
    "当前情绪": {"value": "平静", "type": "enum", "options": ["平静", "喜悦", "悲伤", "愤怒", "紧张"]},
    "关系阶段": {"value": "初识", "type": "enum", "options": ["初识", "朋友", "暧昧", "恋人", "疏远"]}
  }
}
```

**模板 B：冒险**
```json
{
  "genre": "adventure",
  "label": "冒险",
  "vars": {
    "体力": {"value": 100, "type": "int", "min": 0, "max": 100},
    "金币": {"value": 500, "type": "int", "min": 0, "max": 999999},
    "声望": {"value": 0, "type": "int", "min": -100, "max": 100},
    "背包": {"value": [], "type": "list"},
    "队伍": {"value": [], "type": "list"}
  }
}
```

**模板 C：悬疑**
```json
{
  "genre": "mystery",
  "label": "悬疑",
  "vars": {
    "理智值": {"value": 100, "type": "int", "min": 0, "max": 100},
    "线索": {"value": [], "type": "list"},
    "时间进度": {"value": "第1天 上午", "type": "string"},
    "嫌疑度": {"value": 0, "type": "int", "min": 0, "max": 100}
  }
}
```

#### 状态更新规则

LLM 在 `--- 状态 ---` 区块中声明变更，每行一条，格式：`变量名 操作 值`。

**操作符（按变量类型）**：

| 变量类型 | 可用操作 | 格式示例 | 说明 |
|---------|---------|---------|------|
| `int` | `+N` / `-N` | `好感度 +5` / `资金 -200` | 加减数值，结果 clamp 到 `[min, max]` |
| `int` | `=N` | `体力 =100` | 直接设置为指定值（较少用） |
| `enum` | `=值` | `当前情绪 =喜悦` | 设置枚举值，值必须在 `options` 列表中 |
| `string` | `=值` | `时间进度 =第2天 夜晚` | 直接设置字符串 |
| `list` | `+物品名` | `背包 +旧钥匙` | 向列表追加元素（元素已存在则跳过） |
| `list` | `-物品名` | `背包 -旧钥匙` | 从列表移除元素（元素不存在则忽略） |

**程序校验规则**（逐条独立应用，非事务性）：
1. 变量名必须存在于当前模板 → 不在则丢弃该条变更，记日志，并在下一轮 Prompt 中告知 LLM
2. `int` 类型：操作符必须是 `+`/`-`/`=`，值必须是整数；结果 clamp 到 `[min, max]`
3. `enum` 类型：操作符必须是 `=`，值必须在 `options` 列表中 → 不匹配则拒绝该条，并在下一轮 Prompt 中告知 LLM 可用选项
4. `list` 类型：操作符必须是 `+` 或 `-`
5. `string` 类型：操作符必须是 `=`
6. 若 LLM 对 enum/list/string 类型使用了错误的数字操作符（如 `当前情绪 +喜悦`）→ 检测类型不匹配，拒绝，下轮 Prompt 告知

**校验失败的处理**：
- 丢弃该条变更（不影响同轮其他有效变更）
- 记录被拒变更及其拒绝原因到 `rejected_changes` 列表
- 下一轮 Prompt 中注入反馈：`"上一轮以下状态变更被拒绝：当前情绪 +喜悦（enum 类型只支持 =值，可用选项：平静/喜悦/悲伤/愤怒/紧张）"`

#### 状态注入格式

每轮 System Prompt 中注入当前状态摘要。格式：

**int / enum / string 类型**：`变量名 值`，用 `|` 分隔
```
当前状态：好感度 30 | 信任度 45 | 当前情绪 平静 | 关系阶段 朋友
```

**list 类型**：逗号分隔元素，空列表显示为"（空）"
```
当前状态：体力 85 | 金币 500 | 声望 10 | 背包 旧钥匙, 地图 | 队伍 （空）
```

#### 状态影响叙事

- Prompt 中明确要求 LLM：**叙事和可用选项必须与当前状态数值自然结合**
- 好感度低 → 角色对话更冷漠；资金不足 → 某些选项出现但无法选择（通过 `@需要:` 条件标记）
- 选项条件相关的操作符详见 §1.11

### 1.7 结构化输出规范（规范自然语言 + 区块分隔符）

**核心设计决策**：Phase 1 不使用 JSON 或 XML。所有 LLM 输出采用"规范自然语言 + 区块分隔符"格式。LLM 生成自然语言比生成严格 JSON/XML 更稳定，程序用正则按分隔符提取各区块即可。

**通用规则**：
- 区块分隔符格式：`--- 区块名 ---`（顶格，前后无空格）
- 区块之间不要求固定顺序，解析器按区块名独立匹配（顺序无关）
- 区块可多次出现——同名区块内容拼接（如多个 `--- 正文 ---`，但不推荐）
- **正文中禁止**出现单独成行的 `---`（三个短横线），避免被误解析为区块分隔符。若叙事需要分隔线，用其他方式（如 `* * *`）
- LLM 不得在格式区块之外输出任何内容

#### A. 故事设定（共创阶段 LLM → 程序）

**格式示例**：
```
=== 故事设定 ===
题材：romance
世界观：2077年新东京，赛博朋克都市
主角姓名：林晓
主角身份：黑客
主角特质：内向但正义感强
叙事风格：冷峻与温情交织
核心冲突：AI与人类的身份认同危机
主要角色：
  - AIKO | AI助手 | 主角的搭档
  - 田中社长 | 企业总裁 | 幕后黑手
```

**解析规则**：
- 分隔符为 `=== 故事设定 ===`（顶格）
- 顶格字段（`题材：`、`世界观：` 等）→ 正则 `^字段名：(.+)$`
- `题材` 的值必须为 `romance` / `adventure` / `mystery` 之一
- 缩进列表项（`  - ...`）→ 按 `|` 分割为 name / role / relation
- 字段缺失 → 重试（附带缺失字段提示），最多 2 次

#### B. 大纲树（共创阶段 LLM → 程序）

**格式示例**：
```
=== 大纲树 ===
节点1：神秘委托 | ch1_intro
  目标：主角接到一个无法拒绝的黑客任务，初入地下世界
  分支：无

节点2：真相浮现 | ch2_discovery
  目标：发现委托人其实是AI，面临信任危机
  分支说明：根据主角对AI的信任程度走向不同路线
  分支：信任度>=50 → ch3_ally
  分支：信任度<50 → ch3_betrayal

节点3：联手抗敌 | ch3_ally
  目标：与AIKO联手对抗企业
  分支：无

节点4：独自面对 | ch3_betrayal
  目标：独自调查真相，面对内心
  分支：无

节点5：命运交汇 | ch_final
  目标：所有线索汇集，命运在此交汇
  分支：无
```

**字段说明**：
- `节点N：章节标题 | node_id`（必需）— N 从 1 开始递增，node_id 格式 `ch序号_英文缩写`
- `目标：...`（必需）— 本章叙事目标，给 LLM 看的内部指引，不透露给玩家
- `分支说明：...`（可选）— 人类可读的分支方向描述，供 LLM 理解设计意图
- `分支：条件 → node_id`（可选，0-2 条）— 程序可判定的分支条件。无分支时写 `分支：无`

**规则**：
1. 总节点数 5-8 个
2. 每个节点最多 2 个分支
3. 最后一个节点必须写 `分支：无`（结局节点）
4. 分支条件格式：`变量名 运算符 值 → 目标node_id`。运算符见 §1.11
5. 节点标题隐晦，不剧透关键情节
6. 分支条件引用变量名尽可能为当前模板中存在的变量（若不存在，运行时需 LLM 在 `--- 节点 ---` 中补充具体条件）

**静态校验**：
- 所有 `→ 目标node_id` 必须指向大纲中存在的节点
- 若分支条件引用了变量，检查该变量是否存在于当前状态模板
- 校验失败 → 重试（附带错误提示 + 可用变量名列表），最多 2 次

#### C. 叙事输出（游戏循环 LLM → 程序）

##### C.1 基础格式（无分支的普通剧情段）

```
--- 正文 ---
夜幕降临，新东京的霓虹灯在雨雾中晕开一片迷离的光。

你推开地下酒吧的门，潮湿的空气混杂着合成酒精和烧焦电路的气味扑面而来。
吧台深处的角落里，一个戴着全息面具的人朝你招了招手。

--- 选项 ---
1. 径直走过去，开门见山谈正事
2. 先在吧台点一杯酒，观察周围 @需要:理智值>=50
3. 假装没看见，转身离开

--- 状态 ---
理智值 -5
线索 +神秘人的全息面具

--- 续 ---
你走向吧台，全息面具下的目光似乎洞穿了一切……
```

##### C.2 含局部小分支的剧情段（非关键节点）

当剧情中需要小分支（不影响大纲走向的即时选择）时，使用 `--- 分支 ---` 区块：

```
--- 正文 ---
酒吧深处，全息面具人从怀中掏出一枚闪着幽蓝光芒的芯片。

"这就是你要的东西。"他的声音沙哑而低沉，"但要不要接下它，取决于你自己。"

你盯着那枚芯片，脑海中闪过AIKO的警告……

--- 分支 ---
[1] 你伸出手接过芯片。冰凉的金属触感从指尖传来，一道微弱的电流让你打了个激灵——芯片上的数据开始自动解码，全息面具人的嘴角扬起一丝难以察觉的弧度。
[2] 你推开椅子起身，留下一句"我需要时间考虑"，头也不回地走向门口。身后传来芯片在桌面上旋转的细微声响。
[3] 你猛地伸手按住芯片，反手将他的手腕压在吧台上。"先告诉我，AIKO在哪？"酒吧里的音乐戛然而止。

--- 选项 ---
1. 接过芯片
2. 暂时离开
3. 先发制人

--- 状态 ---
理智值 -10

--- 续 ---
酒吧里的气氛骤然紧张起来……
```

**`--- 分支 ---` 区块规则**：
- `[N]` 对应 `--- 选项 ---` 中第 N 个选项（1-based）
- 每个分支片段 50-100 字，简短有力
- 分支数量必须与选项数量一致
- 程序行为：玩家选择选项 N 后，**只展示 `[N]` 片段**，其余分支片段丢弃不展示
- `--- 续 ---` 放在所有 `[N]` 之后，作为通用衔接（即小分支最终汇合）
- 若某个选项不产生分支后果（即选完后自然过渡到 `--- 续 ---`），LLM 可写 `[N] （直接过渡）`

##### C.3 含关键节点的剧情段

当到达大纲关键节点时，`--- 节点 ---` 包含 node_id、可选的分支条件、及 checkpoint 摘要：

```
--- 正文 ---
（到达关键节点的叙事正文……）

--- 选项 ---
1. 和AIKO一起行动
2. 独自行动

--- 状态 ---
信任度 +10

--- 节点 ---
主要节点 ch2_discovery
分支判定：信任度>=50 → ch3_ally
分支判定：信任度<50 → ch3_betrayal
摘要：在酒吧从神秘委托人处获得加密芯片，得知AIKO的真实身份是觉醒的AI。主角选择与AIKO一起潜入企业数据库，发现了"Project Echo"的存在。

--- 续 ---
企业大楼的轮廓在夜空中若隐若现，你和AIKO站在对面楼顶……
```

**`--- 节点 ---` 区块字段**：

| 行 | 必需 | 说明 |
|----|------|------|
| `主要节点 node_id` | ✅ 必选 | 标记到达的关键节点。程序据此推进大纲 |
| `分支判定：条件 → node_id` | 可选 | 当大纲中的分支条件不够具体时，LLM 在此输出运行时条件。若大纲已有足够条件，此行可省略，程序直接用大纲数据评估 |
| `摘要：...` | ✅ 必选 | 从上一 checkpoint 到当前节点的情节概述（纯文本，无格式）。程序存入 checkpoint_summaries。为结局冒险日志提供素材，同时缓解长线游戏中的"早期剧情遗忘"问题 |
| `结束节点` | 结局标记 | 替代"主要节点"，用于故事结局。程序检测到后进入结局流程 |

**`--- 续 ---` 与关键节点**：关键节点分支**不允许在同一个 `--- 续 ---` 中合并**。后续剧情段独立发展，各有各的 bridge text。

##### C.4 无选项的剧情段

当 LLM 判定当前段落不需要玩家选择（过渡/纯叙事）时，省略 `--- 选项 ---`：
- 程序展示正文后自动推进，不等待玩家输入
- 下一轮 User Message 中告知 LLM 玩家"继续推进"（无具体选择）

##### C.5 结局标记

```
--- 结束 ---

--- 冒险日志 ---
（纯文本，面向玩家的冒险回顾）
```

或 LLM 直接返回带结局正文的版本：
```
--- 正文 ---
（结局叙事……）

--- 结束 ---

--- 冒险日志 ---
第一章：神秘委托 —— 主角林晓接到匿名黑客任务，在酒吧与神秘人会面……
第二章：真相浮现 —— 发现委托人AIKO是觉醒AI，面临信任与背叛的抉择……
最终评语：这是一段关于信任与身份的旅程……
```

**结局解析规则**：
- 程序检测到 `--- 结束 ---` → 进入结局流程
- `--- 正文 ---`（如有） → 作为结局叙事展示
- `--- 冒险日志 ---`（如有） → 直接展示给玩家
- 若 LLM 未生成 `--- 冒险日志 ---`，程序在结局流程中额外请求（§1.10-E）

#### D. 区块汇总

| 区块标记 | 阶段 | 必需 | 重复策略 | 说明 |
|----------|------|------|---------|------|
| `--- 正文 ---` | 叙事 | ✅ 必选 | 拼接 | 故事叙述正文。LLM 目标 ~1000 字，自然段空行分隔 |
| `--- 选项 ---` | 叙事 | 可选 | 覆盖（取最后一次） | 2-5 个选项，每行 `N. 描述 @需要:条件` |
| `--- 分支 ---` | 叙事 | 可选 | — | 局部小分支片段，`[N]` 对应选项编号 |
| `--- 状态 ---` | 叙事 | 可选 | 合并 | 每行一条变更：`变量名 操作 值` |
| `--- 节点 ---` | 叙事 | 可选 | 覆盖（取最后一次） | 关键节点标记 + 分支判定 + 摘要 |
| `--- 续 ---` | 叙事 | 可选 | 覆盖（取最后一次） | 下一轮的上文衔接片段 |
| `--- 结束 ---` | 叙事 | 可选 | — | 结局触发标记 |
| `--- 冒险日志 ---` | 结局 | 可选 | — | 面向玩家的冒险回顾文本 |
| `=== 故事设定 ===` | 共创 | 必选 | — | 顶格 `===` 包裹，仅共创阶段使用 |
| `=== 大纲树 ===` | 共创 | 必选 | — | 顶格 `===` 包裹，仅共创阶段使用 |

### 1.8 解析流程与错误处理

#### 1.8.1 解析流程

程序接收 LLM 完整原始响应后执行以下流程（非流式解析——先完整接收，再统一解析）：

```
1. 预处理
   → 去除首尾空白
   → 去除 markdown 代码块围栏（```...``` 或 ```text...```）
   → 检测是共创阶段格式（=== xxx ===）还是叙事阶段格式（--- xxx ---）
      → 共创：按 === 区块名 === 分割
      → 叙事：按正则 ^--- (.+?) ---$ 分割为 (区块名, 内容) 的列表

2. 逐区块解析（顺序无关，按区块名匹配）

   叙事阶段各区块的解析逻辑：
   
   "正文"（可多次出现）：
     → 拼接所有出现的内容（用空行分隔）
     → 即为 narration_text
     → 若为空 → 格式错误，触发重试
   
   "选项"（取最后一次出现）：
     → 逐行解析正则：^(\d+)\.\s+(.+?)(?:\s*@需要:(.+))?$
     → 提取：编号、描述文本、可选条件表达式
     → 解析失败的行 → 严格丢弃
     → 全部失败 → 等同无选项（自动推进）
   
   "分支"：
     → 逐行解析正则：^\[(\d+)\]\s+(.+)$
     → 构建 dict：{选项编号: 分支文本}
     → 玩家选择后按编号查找对应片段展示
   
   "状态"（多次出现则合并所有行）：
     → 逐行解析：变量名 操作符 值
     → 操作符识别：^([+-]\d+)$ = int加减, ^=(.+)$ = 直接设置, ^([+-].+)$ = list增删
     → 逐条送入 GameState.validate_and_apply()
     → 校验失败的条目加入 rejected_changes 列表
   
   "节点"（取最后一次出现）：
     → 第一行：主要节点 <node_id> 或 结束节点
     → 后续行（可选）：分支判定：条件 → node_id
     → 后续行（可选）：摘要：<文本>
     → 验证 node_id 存在于 outline → 不存在则记日志忽略
     → 触发节点推进逻辑（§1.4.4）
   
   "续"（取最后一次出现）：
     → bridge_text = 区块内容（去除首尾空白）
   
   "结束"：
     → 触发结局流程
   
   "冒险日志"：
     → adventure_log_text = 区块内容（直接展示给玩家）

3. 后处理
   → 若 bridge_text 为空且 narration_text 超过长度上限
     → 在上限位置最后一个完整段落处截断
     → 截断点后的内容作为 bridge_text
   → 若截取后 narration_text 低于最低阈值（参考 100 字）
     → 提示用户确认：重试 / 用当前内容继续 / 返回主菜单
```

#### 1.8.2 叙事阶段错误处理

| 场景 | 严重度 | 处理 |
|------|--------|------|
| `--- 正文 ---` 缺失或为空 | 致命 | 重试（附带格式纠正提示），最多 2 次。两次均失败 → 若有上一次有效响应则回退到旧状态；若无（首轮）→ 告知用户，询问重试或返回主菜单 |
| `--- 正文 ---` 中出现 `---` 单独成行 | 致命 | 视为格式错误（LLM 违反规则），重试，最多 2 次 |
| `--- 选项 ---` 缺失 | 正常 | 静默跳过。仅展示正文，不展示选项面板。下一轮自动推进（User Message 标记为无选择） |
| `--- 选项 ---` 中个别行解析失败 | 轻微 | 丢弃该条目，只展示解析成功的选项。若全部失败 → 等同选项缺失 |
| `--- 选项 ---` 中条件引用的变量不存在 | 轻微 | 视为无条件（不阻断选项），记日志 |
| `--- 状态 ---` 中个别行校验失败 | 轻微 | 丢弃该条变更，记录到 rejected_changes，下一轮 Prompt 告知 LLM。不影响其他有效变更 |
| `--- 状态 ---` 中变量名不存在于模板 | 轻微 | 拒绝该条变更，原因记入 rejected_changes |
| `--- 状态 ---` 中 enum 类型使用了错误操作符 | 轻微 | 拒绝，下轮 Prompt 告知正确操作方式 |
| `--- 节点 ---` 中的 node_id 在大纲中不存在 | 中等 | 记日志，忽略该节点标记。不推进大纲，不触发存档。该轮其他处理正常进行 |
| `--- 节点 ---` 中分支条件引用的变量不存在 | 中等 | 记日志，该条件视为无效，跳过。取下一个条件或兜底 |
| `--- 节点 ---` 中缺失摘要 | 轻微 | 记日志，checkpoint_summaries 追加空字符串或节点标题作为 fallback |
| `--- 续 ---` 缺失 | 正常 | 若正文未超长 → bridge_text = ""；若正文超长 → 程序截断 |
| LLM 响应完全无法解析 | 致命 | 重试（附带格式纠正提示），最多 2 次。两次均失败 → 回退到上一次有效状态，告知用户 |
| API 调用失败（网络/rate limit/server error） | 致命 | 告知用户具体错误信息，询问：重试 / 返回主菜单。不做自动重试 |
| 流式输出停顿超时 | 中等 | 截取已生成内容，在最后一个完整段落后截断。截取内容不足阈值 → 提示用户确认 |

#### 1.8.3 共创阶段错误处理

| 场景 | 处理 |
|------|------|
| 故事设定解析失败（字段缺失或格式错误） | 重试（附带缺失字段列表 + 格式纠正提示），最多 2 次。两次均失败 → 告知用户并返回主菜单 |
| 故事设定中 `题材` 不在三选一范围内 | 重试（附带合法值列表），最多 2 次 |
| 大纲树解析失败（格式错误） | 重试（附带格式纠正提示），最多 2 次。两次均失败 → 告知用户并返回主菜单 |
| 大纲树静态校验失败（分支目标节点不存在） | 重试（附带错误提示），最多 2 次 |
| 大纲树静态校验失败（分支条件引用不存在的变量） | 不强制拒绝（LLM 可在运行时补充条件），仅记日志警告 |
| 大纲树节点数 < 3 或 > 15 | 记日志警告（不做强制拒绝，但记录以供调试） |
| 共创阶段 API 调用失败 | 告知用户错误信息，询问：重试 / 返回主菜单 |

#### 1.8.4 重试机制说明

所有需要重试的场景遵循统一模式：
```
def try_with_retry(prompt_fn, parse_fn, max_retries=2):
    for attempt in range(max_retries + 1):
        raw = api_client.call(prompt_fn())
        try:
            return parse_fn(raw)
        except ParseError as e:
            if attempt < max_retries:
                # 修改 prompt_fn，附带错误提示
                prompt_fn = lambda: original_prompt + f"\n[上次格式错误：{e}，请严格按照格式输出]"
            else:
                raise FatalError(f"重试{max_retries}次后仍然失败：{e}")
```

所有 LLM 调用失败不自动重试——告知用户，用户决定。

- **单槽自动存档**：到达关键节点（`--- 节点 --- 主要节点 xxx`）时自动覆盖保存

| 操作 | 触发时机 | 行为 |
|------|---------|------|
| 自动存档 | 程序解析到 `--- 节点 --- 主要节点 xxx` 并完成节点推进后 | 覆盖 `saves/save.json`（原子写入：先写临时文件，成功后 rename） |
| 加载 | 主菜单"继续" | 读取 `saves/save.json` → 验证 JSON 结构完整性 → 恢复全部状态 → 跳入叙事循环 |
| 新建 | 主菜单"新游戏" | 若 `saves/save.json` 已存在 → 提示"已有存档，开始新游戏将覆盖旧存档，确定吗？"；用户确认后覆盖 |
| 存档损坏 | 加载/启动时检测 | 提示"存档已损坏，请开始新游戏"，删除或忽略损坏文件 |

#### 存档损坏判定

以下情况判定为存档损坏（不可恢复）：
- JSON 解析失败（不是合法 JSON）
- `version` 字段缺失
- `story_config` 或 `state_vars` 或 `outline` 或 `progress` 任一关键字段缺失
- `progress.current_node` 指向的 node_id 在 `outline` 中不存在
- `state_template` 指向的模板不存在（模板文件被修改/删除）

存档损坏 → 永久失效，游戏数据丢失。提示用户返回主菜单开始新游戏。不做修复尝试。

#### F. Prompt 中状态注入的格式化函数

以下为程序在组装 Prompt 时应使用的格式化规则：

**state_summary 格式**：
```
int/enum/string 类型：变量名1 值1 | 变量名2 值2 | ...
list 类型：变量名 元素1, 元素2, ...（空列表显示"（空）"）

示例：
好感度 30 | 信任度 45 | 当前情绪 平静 | 体力 85 | 金币 500 | 背包 旧钥匙, 地图 | 队伍 （空）
```

**outline_text 格式**：
按 outline 数组顺序，每个节点输出：
```
- [completed] 节点标题（已完成）
- [active] 节点标题 ← 当前（目标：...）
- [pending] 节点标题
- [pending] 节点标题（可选路线，分支条件：...）
```
对于有分支的非 active/pending 节点，标注分支条件供 LLM 理解设计意图。

**checkpoint_summaries_text 格式**：
```
- ch1_intro（神秘委托）：在酒吧从神秘委托人处获得加密芯片……
- ch2_discovery（真相浮现）：与AIKO一起潜入企业数据库……
```
若尚无摘要，输出"（游戏刚开始，尚无重要事件记录）"。

**rejected_changes_feedback 格式**：
```
上一轮以下状态变更因校验失败被拒绝，请在本次生成中避免重复错误：
- 当前情绪 +喜悦（enum 类型只能用 =值 设置，可用选项：平静/喜悦/悲伤/愤怒/紧张）
- 未知变量 +100（变量名不存在于当前模板）
```
若本轮无拒绝变更，不输出此段。

### 1.11 选项条件系统

#### 格式

LLM 在选项行末尾附带条件标记：
```
2. 先在吧台点一杯酒，观察周围 @需要:理智值>=50
```

**条件表达式格式**：`@需要:变量名 运算符 值`

**支持的运算符**（Phase 1 共 6 个）：

| 运算符 | 适用类型 | 含义 | 示例 |
|--------|---------|------|------|
| `>=` | int | 大于等于 | `@需要:好感度>=30` |
| `<=` | int | 小于等于 | `@需要:嫌疑度<=50` |
| `>` | int | 大于 | `@需要:金币>100` |
| `<` | int | 小于 | `@需要:体力<30` |
| `==` | int, enum, string | 等于 | `@需要:关系阶段==恋人` |
| `has` | list | 列表包含 | `@需要:线索 has 神秘人的全息面具` |

**Phase 1 不支持**：`!=`（不等于）、多条件 AND/OR 组合、括号嵌套。（留到 Phase 2）

#### 程序处理

```
1. 解析选项描述和条件（正则提取 @需要: 标记）
2. 若条件中引用的变量名不存在于当前模板 → 视为无条件，记日志
3. 用当前 GameState 评估条件：
   a. int 类型：用当前值做比较运算
   b. enum/string 类型：用 == 做等值比较
   c. list 类型：用 has 检查列表中是否包含指定元素
4. 满足条件 → 选项状态为"可选"（enabled）
5. 不满足条件 → 选项状态为"置灰"（disabled）
```

#### 展示规则

| 场景 | 展示方式 |
|------|---------|
| 部分选项满足条件 | 满足的选项正常展示；不满足的选项置灰显示（dim 样式），标注 `（需要：变量>=值，当前：值）` |
| 全部选项不满足条件 | **全部正常展示**，移除所有 `@需要:` 条件标注（兜底策略，防止玩家卡死） |
| 无选项 | 不展示选项面板，自动推进 |

#### 示例展示效果

```
  [1] 径直走过去，开门见山谈正事
  [2] 先在吧台点一杯酒，观察周围 （需要：理智值>=50，当前：30）
  [3] 假装没看见，转身离开
```

选项 2 显示但不可选——按键无响应，玩家按 2 时短暂提示"当前条件不满足"。

## Phase 2: 动态系统与增强交互

> Phase 1 稳定后逐步引入。

### 2.1 动态状态定义

- 移除硬编码模板，共创阶段由 LLM 根据题材**自动生成状态变量**
- 变量定义格式与 Phase 1 相同，只是生成方从"预设文件"变为"LLM 输出"
- 需增加额外的**校验步骤**：检查 LLM 生成的变量定义是否合理（变量名不重复、类型合法等）

### 2.2 Web 界面

- 使用 **FastAPI + SSE** 替换 CLI
- 前端：轻量 HTML/JS，支持：
  - 打字机效果流式渲染（真正的逐字 SSE）
  - 选项按钮点击
  - 侧边栏：状态面板 + 日志 + 设置
  - 移动端适配
- Phase 1 的 `game_loop.py` 核心逻辑保留，只替换 `ui.py`

### 2.3 向量记忆

- 引入 `sentence-transformers` 或 API embedding
- 将角色、地点、道具、关键事件 embed 后存 SQLite + `sqlite-vec`
- 每轮检索最相关条目拼入 Prompt
- 作为当前"部分上文"上下文模型的增强——在 System Prompt 中增加"相关背景"区块

### 2.4 多模型支持

- 抽象 LLM 接口层（见 §4.4 原需求）
- 支持为不同任务配置不同模型：
  - 叙事：主力模型（高创意）
  - 审查/追问：便宜模型（低成本）

### 2.5 自定义输入

- 实装 §6.2 的自定义输入管线
- 合理性检查模型判断玩家输入是否越界
- 合理的输入转化为"玩家行动描述"注入 Prompt

### 2.6 一致性检查

- 实现可选的审查者模型（§5.2）
- 检查新剧情与大纲的一致性
- 状态变更合理性验证

### 2.7 存档机制增强

- 多槽位支持（3-5 个槽位，可命名）
- 手动存档（玩家按键触发）
- 存档文件版本兼容（schema 迁移机制）

### 2.8 回档记忆隔离

- 从 checkpoint 回档后，之前路线的对话历史不混入当前上下文
- 保留所有路线的冒险日志用于结局回顾

### 2.9 上下文增强

- 当前 Phase 1 的极简上下文模型（仅上文衔接片段）在长线游戏中可能丢失早期细节
- 引入向量记忆（2.3）作为补充
- 引入可选的历史摘要（仅当上下文明显不足时触发）
- 异步后台生成摘要，不阻塞叙事请求

### 2.10 API 调用容错增强

- 自动重试（可配置次数和间隔）
- 请求失败时的优雅降级策略
- 网络波动时的断点续传
