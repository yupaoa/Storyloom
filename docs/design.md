## 一、项目定位

以 LLM 为叙事大脑、程序为流程管理器的交互式文字冒险游戏引擎。分三阶段交付：

```
Phase 1 ──── CLI 纯文本 MVP ──── 验证核心游戏循环
Phase 2 ──── Web + 动态系统 ──── 从工具迈向产品
Phase 3 ──── 完整体验 ────────── 图像 / 语音 / 云同步
```

### 核心理念

- LLM 是叙事大脑，程序是**流程管理器 + 上下文管家 + 双端接口**。
- 用户与 AI **共创**故事设定和大纲，再进入剧情推进循环。
- 游戏机制极简：剧情导向 + 选项决策，不引入复杂系统。
- 核心挑战：**成本可控、上下文连贯、内容质量稳定、状态随剧情演化**。

---

## 二、整体流程

```
共创阶段（设定 → 大纲）
    ↓
叙事循环（组装 Prompt → LLM 生成 → 解析 → 展示 → 玩家选择 → 状态更新）× N 轮
    ↓
结局（冒险日志 → 回主菜单）
```

### 共创阶段
- 用户与 AI 对话确定题材、世界观、主角与故事方向。**整个过程不透漏具体情节**，保持新鲜度。
- AI 输出结构化的**故事设定**和**大纲树**（有向图，节点 = 关键剧情里程碑，边 = 分支条件）。
- 大纲树不向用户展示，直接进入游戏。确定后不可修改，关键节点即为存档点。

### 叙事循环

每轮 LLM 生成一个"剧情段"，是游戏推进的基本单位。每轮 Prompt 只携带五类信息，不做对话历史累积：

| 信息 | 作用 |
|------|------|
| 大纲树 + 节点进度 | "故事该往哪走" |
| 状态快照 | "过去发生了什么的结果" |
| checkpoint 摘要列表 | "已发生的关键情节" |
| bridge_text | "此刻正在发生什么" |
| 被拒变更反馈 | "上次哪些状态变更不合法" |

核心机制：
- 程序组装 Prompt → LLM 生成剧情段（含结构化区块）→ 程序解析/校验/展示 → 玩家选择 → 状态更新。
- 通过 **bridge 机制**实现无缝衔接：当前段展示结束前，下一段已开始生成。

### 区块分隔符

LLM 输出采用 `--- block_name ---` 结构化标记，程序按正则分割后逐区块解析执行：

| 区块 | 职责 |
|------|------|
| `narrative` | 叙事正文，支持命名分支（`narrative:branch_a`）实现段内变体 |
| `options` | 玩家选项，支持 `@if` 条件置灰和 `-> branch` 路由 |
| `state` | 状态变更建议 + `@branch` 段内路由 |
| `checkpoint` | 大纲节点推进 + `if → route` 大纲分支 |
| `bridge` | 预加载触发点，bridge 之后仅允许 narrative |
| `adventure_log` | 结局冒险回顾（独立 Prompt 生成） |

> 完整语法和路由规则见 [`spec/block-spec.md`](./spec/block-spec.md)。

### 结局
- 大纲终点自然触发，或玩家主动结束（按 Q）。
- 生成**冒险日志**：章节摘要、关键抉择与影响、结局评语，供玩家回味。

## 三、核心设计决策

### 1. 本地为唯一真相源
LLM 只能**建议**变更（状态、节点推进），程序校验后应用。LLM 输出与本地数据冲突时以本地为准。这确保了：
- 数据完整性不受 LLM 幻觉影响
- 存档可脱离原始模型独立恢复
- 选项条件判定、分支路由完全由程序控制

### 2. 程序拥有最终控制权
LLM 负责叙事创意，程序负责流程与数据。API 失败、解析错误、内容异常——告知用户并等待决策，不做自动降级或静默修复。

### 3. 两层分支体系

| 层级 | 范围 | 机制 |
|------|------|------|
| **段内分支** | 当前剧情段内的叙事分叉 | 选项 → 路由到不同 narrative 片段，不影响大纲 |
| **大纲分支** | 故事走向分叉 | checkpoint 中条件判定 → 路由到不同后续节点 |

两层解耦，段内分支提供局部交互深度，大纲分支决定全局走向。

### 4. bridge 机制
剧情段分为**交互区**（options / state / checkpoint）和**缓冲区**（bridge 之后至段末的叙事）。程序读到 bridge 时立即触发下一轮 Prompt 组装，同时继续展示缓冲文本——用户体感连续，不感知段边界。

## 四、状态系统设计

**设计目标**：状态系统随故事而变——每个故事有自己的变量集，由 LLM 在共创阶段根据故事设定自动设计。

**变量定义**：共创阶段 Step 3.5 中，LLM 根据已确认的 story_config 生成 5–8 个变量。约束：

- **number**：范围统一 [0, 100]，初始值须在范围内
- **string**：自由文本，替代传统枚举类型，初始值非空
- **list**：元素为 string（如背包、线索、技能），初始值可为空数组

程序逐条校验：变量名合法/唯一、类型仅限三种、初始值合规。校验失败 → 重试（附带错误提示）。

**运行时**：每轮 LLM 在 `--- state ---` 中附带变更指令（加减数值、设置字符串、增删列表元素），程序逐条校验——变量存在、类型匹配、number 范围 clamp [0,100]——通过后方可应用。被拒绝的变更记入 `rejected_changes`，下一轮 Prompt 中告知 LLM。

**影响叙事**：Prompt 要求 LLM 将当前状态自然融入叙事和选项。

## 五、上下文与成本策略

Phase 1 以 checkpoint 摘要 + bridge_text 维持上下文，不做向量检索或压缩式记忆。

- **抽象 LLM 接口**：支持 OpenAI 兼容 API。Phase 1 使用单一模型，预留多模型路由扩展点。
- **System Prompt 缓存**：利用 API 缓存固定前缀（大纲、设定），降低重复计费。
- Phase 2+ 计划：向量记忆、分层压缩、多模型路由（叙事/审查分用不同模型）。

## 六、质量保障哲学

### 结构化输出
LLM 输出采用**区块分隔符**（而非 JSON/XML）——自然语言比严格格式更稳定，程序用正则提取。格式解析失败则重试（附带纠正提示），耗尽后交用户决策。

### 校验分层

| 层级 | 处理 |
|------|------|
| 格式错误（区块解析失败） | 重试，耗尽告知用户 |
| 数据错误（引用不存在的变量/node_id） | 拒绝该条，记入反馈，下轮告知 LLM |
| 微小偏差（数值越界、重复操作） | 静默修正，不中断流程 |

### 大纲锚定
大纲节点标注完成状态（completed / active / pending），当前章节目标注入 Prompt。节点推进由程序根据 checkpoint 中的条件判定结果执行，LLM 不自主决定大纲走向。"基准有约束，细节可灵动"。

## 七、交互设计

- **固定选项为主**：每轮 0-4 个选项（按故事档位），附带条件限制和后果暗示。全部不可用时兜底展示，防止卡死。
- **自定义输入为辅**（⚠️ Phase 2+）：玩家输入自由文本，经合理性检查后转化为规范行动描述。
- **展示节奏**：打字机效果逐段输出，支持自动/手动推进切换（按 M）。
- **界面**（⚠️ Phase 2+ Web）：Phase 1 为终端 CLI。

---

## 八、存档设计

- **自包含**：存档 = 完整游戏状态快照（设定含 variables、大纲、状态变量、checkpoint 历史）。变量定义存储于存档内部，不受未来变量系统变更影响。
- **模型无关**：存档不绑定模型标识——切换模型后重新注入 Prompt 即可继续。
- **原子写入**：先写临时文件（`.tmp`），再 `os.replace`，防止写入中断导致损坏。
- **自动存档**：仅在 checkpoint 到达时触发，不设手动存档。
- **完整性校验**：加载时校验版本号、关键字段完整性（story_config 含 variables、state_vars、outline、progress），任一失败则判定存档损坏。
- ⚠️ Phase 2+：checkpoint 回档、云同步。

## 九、扩展路线

### Phase 1 — CLI 纯文本 MVP（当前）
终端 CLI、LLM 自定义变量、共创阶段变量定义、固定选项、自动存档。验证核心游戏循环。

### Phase 2 — Web + 动态系统

| 能力 | 变化 |
|------|------|
| **变量系统增强** | 可用变量数扩展至 10+，支持更复杂的数值约束和依赖关系 |
| **Web 界面** | FastAPI + SSE 流式渲染，选项按钮、状态面板、移动端适配 |
| **向量记忆** | 角色/地点/事件 embed 存储，每轮检索注入 Prompt |
| **多模型** | 叙事用主力模型，审查/追问用便宜模型，降本增效 |
| **自定义输入** | 玩家可自由输入行动，合理性检查模型把关 |
| **一致性审查** | 独立模型检查剧情与大纲、状态变更的合理性 |

Phase 1 核心逻辑保留，仅替换 UI 层。

### Phase 3 — 完整体验
在 Phase 2 成熟后按需启动：
- **图像生成** — 关键场景异步生成插画
- **云同步** — 存档加密上传，跨设备同步
- **TTS** — 可选角色语音朗读
- **剧本导出** — 冒险历史格式化为 Markdown/PDF
- **多人模式** — 不同玩家扮演不同角色，AI 居中叙述协调

## 十、设计原则

| 原则 | 说明 |
|------|------|
| **LLM 叙事，程序控流** | LLM 负责创意内容；程序负责数据完整性、流程控制、状态校验 |
| **本地唯一真相源** | 一切游戏数据以本地 GameState 为准，LLM 只能建议变更 |
| **预加载无缝衔接** | bridge 机制确保下一段在用户阅读当前段时已后台生成 |
| **失败透明** | API 错误、解析失败告知用户并等待决策，不静默回退 |
| **错误隔离** | 单条 state 失败不影响同段其他变更；单选项解析失败不影响其余选项 |

---

## 十一、非功能性需求

- **可扩展**：状态系统、记忆策略、模型接口均设计为可插拔模块。
- **容错**：API 失败 → 告知用户决策；解析失败 → 有限重试后告知用户；数据校验 → 逐条隔离，单条失败不影响其余。
- **隐私**：API Key 本地存储，不经过第三方。
- **响应**：流式输出 + bridge 预请求，减少用户等待感知。
