# 项目总体需求：Storyloom: AI 交互式文字小说游戏引擎

## 一、核心理念
- 以 LLM 作为叙事大脑，程序作为“流程管理器 + 双端接口 + 上下文管家 + 交互界面”。
- 支持从零开始与用户共创故事设定和大纲，再进入剧情推进循环。
- 游戏机制极简：剧情导向 + 固定选项，不引入复杂战斗或模拟系统。
- 核心挑战是：**成本可控、上下文连贯、内容质量稳定、状态系统随剧情而变**。

---

## 二、整体游戏流程

### 1. 共创阶段（设定 & 大纲）
- 用户先与 AI 对话，明确题材、世界观、主角、大致故事方向。
  - 可由用户提供原始要求，AI对关键设定进行多次询问，确定决策树
  - 整个过程不应当透露具体情节，保持内容新鲜度
- 程序将讨论要点归纳为**结构化的故事设定**（JSON），包括：
  - 题材标签（恋爱/冒险/经营/悬疑…）
  - 核心人物、关系、场景
  - 主要故事风格和走向
  - 关键冲突与高潮节点
- 设定确认后，AI 生成**大纲树**：**主干情节 + 状态系统 + 关键节点**，节点标注分支方向（可只有一个分支），但不写死所有细节。
- 大纲树**不需要**获得用户确认，直接进入游戏。

注意：
- 大纲树确定后不得进行修改，关键节点即为存档点，游戏不设置自定义存档位置
- 分支节点数量不宜过多，以减轻程序负担
- 分支节点需要预先设定，但可以在流程中判定分支，可能依据选项、好感度等各种因素

### 2. 游戏进行中（叙事循环）
- 程序将当前**大纲树 + 剧情进度 + 状态系统状态 + 关键说明 + 部分上文情节**组装成 Prompt
- 请求 LLM 合理生成后续剧情，每轮**格式化**生成一个合理长度的**剧情段**，程序逐句展示
- 在合适的剧情段位置可以设置选项，选项需格式化输出，包含在剧情段内，由程序进行处理
- 选项可能包含状态变更（如好感度变化、获得物品、时间推进等），程序解析后更新游戏状态
- 剧情段分为两种：不包含关键节点（长），包含关键节点（短）
- 若包含关键节点，则会在执行到节点处时进行节点判定，并更新剧情和存档进度
- 界面展示文字、选项，玩家做出选择
- 当本轮剧情段**临近结束**时，打包为下一轮的输入，继续循环

注意：
- 选项可以是简单的剧情局部分支，提高游戏沉浸感，不要求一定产生数据层面的影响，但不宜过多
- 每个剧情段包含的关键节点不宜超过一个
- 程序执行到后续没有关键节点的尾部片段时，就应当把后续内容作为部分上文情节，组装Prompt进行下一轮，保证无缝衔接
- 若包含分支节点，则需要预先生成后续所有分支的情节（部分），在执行到分支时判定
- 若包含分支节点，则建议剧情段末尾结束在分支后合理位置，兼顾生成剧情总量和后一轮的衔接

### 3. 结局与回顾
- 当大纲标记的结局条件达成（或用户主动结束），游戏进入尾声。
- 系统可生成一份**冒险日志**：根据记忆记录整理出章节摘要、关键决策、状态变化曲线，供玩家回味。
- 游戏历经过的关键节点都允许用户回档，以回档点为起点进行新一轮游戏（状态系统需匹配），不要求剧情文本一致

## 三、动态状态系统（随题材而变）

状态不硬编码，而是在游戏开始时由 LLM 根据题材定义，并在过程中动态维护。

### 1. 状态定义阶段
- 在共创阶段末尾，程序要求 LLM 根据确定的故事设定，输出一个**状态模板**，例如：
  - 恋爱向：`好感度:0`, `信任度:0`, `当前情绪:平静`
  - 经营向：`资金:1000`, `声誉:50`, `库存:[]`, `雇员:[]`
  - 恐怖悬疑：`理智值:100`, `线索收集:[ ]`, `时间倒计时:72h`
- 状态模板被存储为一组带数据类型的键值对，并可定义初始值及可选范围。

### 2. 状态更新机制
- 每一轮 LLM 生成回应时，如有必要，需在返回的结构化数据中附带**状态变更指令**，如：
  ```json
  "state_changes": [
    {"var": "好感度", "op": "add", "value": 5},
    {"var": "资金", "op": "sub", "value": 200}
  ]
  ```
- 程序接收后严格更新内部状态存储，不允许模型直接篡改全局记忆，确保数据可信。
- 若 LLM 输出的变更涉及未定义变量，程序可自动拒绝。
- 状态会以简洁形式注入下一轮 Prompt，例如：“当前状态：好感度30，资金800，持有道具：旧钥匙”。

### 3. 状态影响叙事
- Prompt 中明确要求 LLM：**叙事和可用选项必须与当前状态和数值自然结合**。
- 例如好感度低时，角色对话更冷漠；资金不足时，某些选项变灰或产生相应后果。
- 选项本身可以附带“所需状态条件”，程序据此过滤或高亮选项，增强决策感。

## 四、上下文与成本管理

上下文管理与成本控制是本项目的核心工程难题，需要在长期迭代中逐步完善。以下策略共同服务于一个目标：**在有限的 token 预算内，维持尽可能高质量的叙事连贯性**。特别地，§2 中"预生成分支情节"的机制会对 token 消耗产生额外压力，需要在上下文调度时予以特殊考虑。

### 1. 分层记忆架构
- **即时上下文**：保留最近邻接的上文情节，确保剧情连贯。
- **压缩式长期记忆**：
  - 当历史轮次超过阈值，调用 LLM（或专用小模型，见 §4.4 多模型配置）对更早的对话生成**逐章摘要**和**重要事件列表**。
  - 摘要持久化存储，并作为系统提示的一部分始终保留。
- **关键信息检索库（向量记忆）**：
  - 将角色、地点、道具、关键事件等实体信息向量化存储。
  - 每轮生成前，用当前文本做相似度检索，召回最相关的背景条目（如“这个人之前与主角发生过什么”），拼入 Prompt 的显式记忆块。
  - 这能低成本地模拟长文本记忆，远优于无限堆叠历史。

### 2. Token 预算控制
- 程序需估算每次请求的 token 数量，动态调整：
  - 历史轮次过多时，自动触发摘要压缩。
  - 若检索到的记忆条目过多，按相关度截断。
  - 允许用户选择模型（如上下文窗口大小不同），程序自动适配最大长度。
- 对于付费 API，可设置单次最大 token 限额，超出时优先精简描述性文字或缩短选项说明。
- **分支预生成的预算**：当 §2 中触发分支节点预生成时，程序应将多个分支的生成请求纳入统一的 token 预算管理，避免单轮成本失控。可考虑的策略包括：降低分支预生成的长度、使用更便宜的模型生成分支草稿、或限制同时预生成的分支数量。

### 3. 缓存与复用
- 对固定的大纲描述、世界设定、状态模板等不常变部分，采用固定前缀的方式，不必每次重新生成。
- 若使用本地模型，可用推理缓存；API 侧可利用系统提示缓存（如有支持）。

### 4. 多模型可选配置
- 抽象出一层通用 LLM 接口，支持：
  - OpenAI 兼容 API（可自定义 endpoint，适用多数代理、本地模型）
  - 特定商业 API（Claude、Gemini 等，由适配器实现）
- 用户可全局配置模型参数（temperature、top_p 等），并针对“叙事”和“检查合理性”这类不同任务分配不同模型（如创意性任务用大模型，合理性检查用便宜的小模型）。

## 五、内容质量保障体系

### 1. 结构化输出强制
- 所有 LLM 响应必须遵循严格格式，程序才能可靠解析。推荐使用 XML 标签包裹各语义块，其中 `state_changes` 内嵌 JSON（与 §3.2 格式一致）。示例：
  ```
  <narration>
    故事叙述段落……
  </narration>
  <options>
    - 选项1描述 (条件: 好感度>=20)
    - 选项2描述
  </options>
  <state_changes>
    [
      {"var": "好感度", "op": "add", "value": 5},
      {"var": "资金", "op": "sub", "value": 200}
    ]
  </state_changes>
  <checkpoint type="major" node_id="ch3_climax" />
  ```
- 各标签说明：
  - `<narration>`：必选，本段故事叙述正文
  - `<options>`：可选，包含 2-5 个选项，每个选项可附带所需状态条件
  - `<state_changes>`：可选，JSON 数组，变更格式见 §3.2
  - `<checkpoint>`：可选，标记当前是否到达关键节点及其类型
- 如果解析失败，自动重试（附带错误提示）或回退到上一次有效状态。

### 2. 一致性检查机制
- 可选的”审查者”模型：在每一轮生成后，额外调用一个轻量 LLM（或同一模型的不同调用），检查：
  - 新剧情是否与大纲、人物设定矛盾？
  - 状态变更是否合理？
  - 选项是否重复或不合逻辑？
- 若发现问题，自动将反馈插入下一轮的系统提示中进行修正，或请求重新生成。
- 该审查机制与 §6.2 中自定义输入的”合理性检查”可共用同一检查管线，仅在检查对象（LLM 生成内容 vs 用户输入）和失败处理策略上有所区别。

### 3. 大纲锚定与灵活度
- 将大纲拆分为“已完成章节 + 当前章节目标 + 未来强制节点”。
- 每次请求前，将当前章节目标前置：“本章目标是让主角与神秘人相遇并发现密室，请推动剧情朝此发展，但具体的对话和环境由你发挥。”
- 章节结束时，系统触发进度检查：LLM 自我判断目标是否达成，未达成则延长当前章节或调整大纲。
- 这样保证了“基准有约束，细节可灵动”。

### 4. 自动纠错与回滚
- 如果玩家明显不满或输出明显崩坏，用户可以手动撤回并选择稍有不同的行动，程序需支持完整的操作历史回溯。

## 六、用户交互与选项系统

游戏交互以**固定选项为主、自定义输入为辅**。固定选项保证叙事引导和节奏控制；自定义输入提供自由度，但须经过合理性检查以防止脱离故事框架。

### 1. 固定选项生成规范
- 每轮 LLM 必须生成 3-5 个选项，格式明确。
- 选项可以附带简短的后果暗示（”（可能会激怒对方）”），但应保持叙事内视角，避免打破第四面墙。
- 选项应具有信息差和决策价值，避免全部导向同一结果。
- 程序可以配置“选项多样性”策略，比如禁止连续出现语义重复的选项。

### 2. 自定义输入处理
- 在任何时候，玩家都可以输入自由文本，而不是点击选项。
- 处理流程：
  1. 用户输入文本
  2. 调用”合理性检查”模型（可复用 §5.2 的审查管线），结合当前状态、上下文、大纲约束，判断该行动：
     - 是否在故事世界里合理？
     - 是否会严重破坏主线？
     - 若合理，则转化为规范的”玩家行动描述”；若不合理，生成友好提示并建议修改。
  3. 将转化后的行动作为下一轮 LLM 的输入，同时记录该动作为玩家选择。
- 此举在保持自由度的同时，守住剧情底线，不使故事脱缰。

### 3. 界面交互需求
- 基本布局：主要叙事区（大段文字，可滚动）+ 选项面板 + 侧边栏（状态查看、日志、设置）。
- 支持打字机效果输出文字，增强沉浸感。
- 选项出现时可带微小延迟，营造思考感。
- 提供“查看完整对话记录”和“查看当前状态”按钮。
- 移动端优先，但桌面也需适配。

## 七、图像生成集成（未来扩展）

首批版本不实现，但架构需预留扩展点。

- **异步生成插画**：当剧情推进到重要场景或用户主动触发，后台调用图像模型（Stable Diffusion / DALL·E）基于当前场景描述生成插画。
- 图片插入到对话历史中，形成”视觉小说”感。
- 支持配置生成风格、是否自动生成、每日/每回合生成上限。
- 图像生成请求独立于叙事请求，不阻塞主游戏循环。

---

## 八、存档与持久化

- 存档文件包含：故事设定、大纲树、完整状态快照、记忆摘要、向量索引元数据（向量嵌入可选择重新生成或随档存储，取决于存储成本与精度要求的权衡）、对话历史（或摘要后的精简版）、模型配置。
- 支持多存档槽位，命名备注。
- 存档应独立于模型，即使切换模型，基于相同状态也能继续游戏（只需重新注入 Prompt）。注意：若切换的模型嵌入维度不同，向量索引需重建。
- 可选云同步，但初期以本地 JSON/二进制文件为主。

---

## 九、非功能性需求

- **可扩展性**：状态系统、记忆策略、模型接口均需设计为可插拔模块，方便后续加入新功能（如战斗系统、多人在线协作讲故事等）。
- **隐私与安全**：用户 API key 本地加密存储，不经过任何第三方服务器（如果产品是本地应用）。若联网，需明确隐私政策。
- **响应速度**：优化 token 使用以减少等待；支持流式输出以驱动打字机效果（见 §6.3），给用户即时反馈感。
- **容错**：API 调用失败时自动重试并降级（如使用备选模型）；结构化输出解析失败时提供手动重试选项。

---

## 十、未来可能的增强方向（非当前必须）

- 剧本导出功能（将冒险历史格式化为小说或剧本格式）。
- 多人模式：不同玩家扮演不同角色，AI 居中叙述。
- 语音合成朗读（TTS）。
- 调试模式：开发者可查看实际发送给 LLM 的完整 Prompt，便于 tweak。
