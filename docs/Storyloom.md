# Storyloom: AI 交互式文字小说游戏引擎

> **定位**：设计理念与架构愿景（长期参考）。  
> **实施**：Phase 1 纯文本 CLI MVP，以 [`Phase1-exec-flow.md`](./Phase1-exec-flow.md) 为权威执行文档。  
> 标注 `⚠️ Phase 2+` 的内容尚未实现。
>
> **最后更新**：2026-07-04

---

## 一、核心理念

- LLM 是叙事大脑，程序是**流程管理器 + 上下文管家 + 双端接口**。
- 用户与 AI **共创**故事设定和大纲，再进入剧情推进循环。
- 游戏机制极简：剧情导向 + 选项决策，不引入复杂系统。
- 核心挑战：**成本可控、上下文连贯、内容质量稳定、状态随剧情演化**。

---

## 二、整体流程

```
共创阶段（设定 → 大纲）
    ↓
叙事循环（组装 Prompt → LLM 生成 → 解析 → 展示 → 玩家选择 → 状态更新）× N 轮
    ↓
结局（冒险日志 → 回主菜单）
```

### 共创阶段
- 用户与 AI 对话确定题材、世界观、主角与故事方向。**整个过程不透漏具体情节**，保持新鲜度。
- AI 输出结构化的**故事设定**和**大纲树**（有向图，节点 = 关键剧情里程碑，边 = 分支条件）。
- 大纲树不向用户展示，直接进入游戏。确定后不可修改，关键节点即为存档点。

### 叙事循环
- 每轮将大纲进度 + 状态快照 + 上文衔接文本组装为 Prompt，LLM 生成一个**剧情段**。
- 剧情段内包含正文、选项、状态变更建议、节点推进标记等**结构化区块**。
- 程序解析区块、校验数据、展示内容、收集玩家选择，更新状态后进入下一轮。
- 通过 **bridge 机制**实现无缝衔接：当前段展示结束前，下一段已开始生成。

### 结局
- 大纲终点自然触发，或玩家主动结束。
- 生成**冒险日志**：章节摘要、关键抉择与影响、结局评语，供玩家回味。

## 三、核心设计决策

### 1. 本地为唯一真相源
LLM 只能**建议**变更（状态、节点推进），程序校验后应用。LLM 输出与本地数据冲突时以本地为准。这确保了：
- 数据完整性不受 LLM 幻觉影响
- 存档可脱离原始模型独立恢复
- 选项条件判定、分支路由完全由程序控制

### 2. 程序拥有最终控制权
LLM 负责叙事创意，程序负责流程与数据。API 失败、解析错误、内容异常——告知用户并等待决策，不做自动降级或静默修复。

### 3. 两层分支体系
| 层级 | 范围 | 机制 |
|------|------|------|
| **段内分支** | 当前剧情段内的叙事分叉 | 选项 → 路由到不同 narrative 片段，不影响大纲 |
| **大纲分支** | 故事走向分叉 | checkpoint 中条件判定 → 路由到不同后续节点 |

两层解耦，段内分支提供局部交互深度，大纲分支决定全局走向。

### 4. bridge 机制
剧情段分为**交互区**（options / state / checkpoint）和**缓冲区**（bridge 之后至段末的叙事）。程序读到 bridge 时立即触发下一轮 Prompt 组装，同时继续展示缓冲文本——用户体感连续，不感知段边界。

## 四、状态系统设计

> ⚠️ Phase 2+：状态由 LLM 在共创阶段动态定义。Phase 1 使用三套硬编码模板（恋爱/冒险/悬疑）。

**设计目标**：状态系统随题材而变——恋爱向追踪好感度与信任度，悬疑向追踪理智值与线索，经营向追踪资金与库存。

- **定义**：共创末尾，LLM 输出一组带类型的键值对（初始值 + 可选范围）。
- **更新**：每轮 LLM 在结构化输出中附带变更指令，程序逐条校验后应用。引用未定义变量 → 拒绝；数值越界 → clamp；重复操作 → 静默忽略。
- **影响叙事**：Prompt 要求 LLM 将当前状态自然融入叙事和选项（好感度低 → 角色冷漠；资金不足 → 选项置灰）。

## 五、上下文与成本策略

> 本章主要为 Phase 2+ 愿景。Phase 1 以 checkpoint 摘要 + bridge_text 维持上下文，不做向量检索。

### 分层记忆
- **即时上下文**：最近邻接情节，确保叙事连贯。
- **压缩式长期记忆**：历史超阈值时，LLM 生成逐章摘要和重要事件列表，持久化注入 System Prompt。
- **向量记忆**：角色、地点、道具等实体向量化存储，每轮检索最相关背景条目——低成本模拟长文本记忆。

### Token 预算
- 估算每次请求 token 数，超限时触发摘要压缩或截断检索结果。
- 固定前缀（大纲、设定）利用 System Prompt 缓存，避免重复计费。

### 多模型路由
- 抽象通用 LLM 接口，支持 OpenAI 兼容 API 及商业 API。
- 不同任务可分配不同模型：创意叙事用大模型，合理性检查用便宜小模型。

## 六、质量保障哲学

### 结构化输出
LLM 输出采用**区块分隔符**（而非 JSON/XML）——自然语言比严格格式更稳定，程序用正则提取。格式解析失败则重试（附带纠正提示），耗尽后交用户决策。

### 校验分层
| 层级 | 处理 |
|------|------|
| 格式错误（区块解析失败） | 重试，耗尽告知用户 |
| 数据错误（引用不存在的变量/node_id） | 拒绝该条，记入反馈，下轮告知 LLM |
| 微小偏差（数值越界、重复操作） | 静默修正，不中断流程 |

### 大纲锚定
大纲节点标注完成状态（completed / active / pending），当前章节目标注入 Prompt。节点推进由程序根据 checkpoint 中的条件判定结果执行，LLM 不自主决定大纲走向。"基准有约束，细节可灵动"。

## 七、交互设计

- **固定选项为主**：每轮 0-4 个选项（按故事档位），附带条件限制和后果暗示。全部不可用时兜底展示，防止卡死。
- **自定义输入为辅**（⚠️ Phase 2+）：玩家输入自由文本，经合理性检查后转化为规范行动描述。
- **展示节奏**：打字机效果逐段输出，支持自动/手动推进切换。
- **界面**（⚠️ Phase 2+ Web）：Phase 1 为终端 CLI。

---

## 八、存档设计

- **自包含**：存档 = 完整游戏状态快照（设定、大纲、状态变量、checkpoint 历史）。与模板文件解耦，模板更新不影响已有存档。
- **模型无关**：存档不绑定模型标识——切换模型后重新注入 Prompt 即可继续。
- **原子写入**：先写临时文件，再 rename，防止写入中断导致损坏。
- **自动存档**：仅在 checkpoint 到达时触发，不设手动存档。
- ⚠️ Phase 2+：checkpoint 回档、云同步。

## 十、非功能性需求

- **可扩展**：状态系统、记忆策略、模型接口均设计为可插拔模块。
- **容错**：API 失败 → 告知用户决策；解析失败 → 有限重试后告知用户；数据校验 → 逐条隔离，单条失败不影响其余。
- **隐私**：API Key 本地存储，不经过第三方。
- **响应**：流式输出 + bridge 预请求，减少用户等待感知。
