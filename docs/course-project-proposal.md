## 1 项目名称

**Storyloom** — AI 驱动的交互式文字小说游戏引擎

## 2 一句话简介

Storyloom 是一个让 AI（大语言模型）担任"说书人"的文字冒险游戏引擎——程序负责管理游戏状态、控制叙事流程、校验 AI 输出，AI 负责实时生成故事内容并与玩家互动，二者配合实现"无限分支、千人千面"的交互式小说体验。

## 3 选题说明

> **选题由来**：本项目并非课程作业的产物，而是个人一直以来的想法——上学期考完试后开始尝试，从纯文本的设计构想逐步演化为明确的工程方向。最早还构想了带图片生成的完整模式，后合理划分阶段，将当前目标锁定为 Phase 1 核心引擎。

### 3.1 项目做什么

玩家进入游戏后，先通过"共创阶段"与 AI 对话确定故事设定（题材、世界观、角色、关键变量），AI 据此生成故事大纲（有向图结构）。随后进入核心叙事循环：

1. AI 实时生成一段带选项的故事文本（XML 结构化输出）
2. 程序**流式解析** AI 输出，在合适时机预取下一轮 Prompt，实现无缝连续叙事
3. 玩家做出选择 → 程序更新游戏状态 → 进入下一轮

项目完整实现了一个**文本游戏引擎**，包含：流式 XML 解析器、对话式上下文管理器（滑动窗口 + 压缩）、状态校验系统、原子化存档管理、结局检测、国际化（中英文）等子系统。

### 3.2 课程知识点体现

**《程序设计》相关：**

| 知识点 | 项目中的应用 |
|--------|-------------|
| **I/O 流** ⭐ | **流式输出**：接收 LLM 的 HTTP 流式响应，边接收边解析，实现故事文本的实时逐字显示，避免玩家长时间等待；**文件读写**：使用 Python 文件操作保存/读取游戏进度（JSON 格式），并采用"写临时文件 → 原子替换"策略保证存档安全 |
| 异常处理 | API 调用超时、JSON 解析失败、XML 格式错误等异常分层捕获，每类错误有明确的恢复或提示路径 |
| 字符串处理 | XML 输出逐行解析、文本内容提取与拼接、中英文国际化字符串替换 |
| 函数与模块化 | 引擎按功能拆分为独立模块（游戏循环、解析器、API 客户端、配置），模块间通过明确的函数接口协作 |
| 循环控制 | 初期使用递归和 `while True` 导致系统崩溃（WSL 资源耗尽），后改为有限循环 + 迭代计数器，理解边界条件的重要性 |
| 面向对象 | `GameState`、`ContextManager`、`StreamingXmlParser` 等核心组件封装为类，通过对象组合构建系统 |
| 列表与字典 | 游戏状态变量（dict）、对话历史消息列表（list）、选项字典（dict）贯穿整个引擎 |

**《数据结构》相关：**

| 知识点 | 项目中的应用 |
|--------|-------------|
| 有向图 | 故事大纲建模为 checkpoint 节点 + route 边的有向图，支持分支路由与条件跳转 |
| 栈 | XML 标签嵌套匹配（`<choice>`→`<opt>`→`</choice>`、`<branch>`→`<seg>`→`</branch>`），开标签入栈、闭标签出栈，保证嵌套层级正确 |
| 队列 | 使用 `queue.Queue` 实现 bridge 预取机制——后台线程异步请求下一轮，前台继续展示，解耦生成与消费速度 |
| 滑动窗口 | 上下文管理器维护对话历史的固定大小窗口（最近 N 轮完整保留，早期轮次压缩为摘要），控制内存占用 |
| 树形结构 | XML 解析结果建模为嵌套 dataclass 树（`StoryOutput → SegBlock → ChoiceBlock → Option`），递归遍历收集内容 |
| 哈希表 | 状态变量、选项字典、分支路由均使用 dict 实现 O(1) 查找 |

### 3.3 实际应用价值

- **AI Native 应用开发**：探索 LLM 作为内容引擎的工程实践——如何约束 AI 输出、如何管理上下文窗口、如何在"AI 创造力"和"程序确定性"之间取得平衡
- **游戏引擎设计**：研究叙事驱动型游戏的引擎架构，解决流式内容生成、状态管理、分支路由等核心问题
- **可扩展平台**：引擎核心与 UI 分离，同一引擎可驱动终端 CLI、Web 界面、移动 App 等多种前端

### 3.4 技术难度

| 挑战 | 说明 |
|------|------|
| LLM 输出约束 | 设计 XML 输出格式规范，用 Prompt Engineering 让 AI 严格遵守，需大量实验迭代 |
| 流式解析 + 预取 | 在流式接收 AI 输出的同时，在合适时机（bridge 标记）提前发起下一轮请求，压缩感知延迟 |
| 上下文窗口管理 | AI 的上下文窗口有限（约 128K tokens），需要滑动窗口 + checkpoint 压缩策略在"记忆"与"成本"间平衡 |
| 状态校验 | AI 输出不可信——程序必须校验每个状态修改建议的类型、范围、变量存在性，拒绝非法操作并反馈 |
| 双人协作架构 | 引擎与 UI 分层开发，需设计清晰的 API 边界和文件管辖规则，避免合并冲突 |

### 3.5 团队协作与个人工作

项目采用**引擎 + UI 双人并行开发**模式。

**本人（引擎开发者）负责：**
- 游戏循环与状态管理系统（GameLoop、GameState）
- 流式 XML 解析器（StreamingXmlParser）
- LLM 交互层（API 客户端、Prompt 构建、上下文管理）
- 存档系统（原子 JSON 读写）
- 共创流程（Co-Create：Q&A → 设定 → 大纲生成）
- 国际化框架（gettext 中英文支持）
- 全部单元/集成测试（227 个测试用例）
- 全部技术规格文档（4 篇权威 spec + 工程日志）

**UI 开发者负责：**
- Web 界面（FastAPI + SSE 流式渲染）
- 交互设计与状态展示面板

协作规则：引擎核心文件 UI 层只读不写；UI 文件引擎层不依赖。通过 `GameSession` 统一 API 接口连接，两个方向独立迭代、互不阻塞。

## 4 当前开发进度

- **Phase 1 核心引擎**：已完成 ✅

- **Web 界面**：并行分支活跃开发中（FastAPI + SSE）
- **后续规划**：向量记忆、自定义输入、一致性审查、多模型调度、图像生成等

## 5 总结

Storyloom 是一个**来源于实际兴趣、面向真实应用场景**的项目——它不仅仅是一个"课程大作业"，而是有明确工程目标、可持续演进的软件系统。项目涵盖程序设计（**I/O 流**为核心亮点，配合异常处理、字符串处理、函数与模块化、循环控制、面向对象、列表与字典）和数据结构（图、栈、队列、滑动窗口、树、哈希表）两大课程的核心知识点，同时涉及 AI 应用开发、系统工程、团队协作等前沿实践，适合作为编程综合实践的选题。
